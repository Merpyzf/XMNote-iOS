# 阅读页回弹异常复盘

评估日期：2026-04-12

## 1. 结论摘要
- 受害者不是 `ReadingDashboardView` 或 `ReadingTimelineView` 自己，而是阅读 Tab 根页共享的滚动宿主。
- 真正 owner 是 `MainTabView` 上根级搜索宿主的挂载范围，也就是 `searchable + tabViewSearchActivation` 挂在了过大的共享容器上。
- `BookSearchView` 能把异常“治好”，不是因为它路径天然正确，而是因为它在同一条 `readingPath` 内通过 `.searchable(... placement: .navigationBarDrawer(displayMode: .always))` 触发了系统搜索宿主重绑。

## 2. 现象与复现矩阵

| 场景 | 结果 | 说明 |
| --- | --- | --- |
| 冷启动后在读首页下拉 overscroll | 正常 | 初始宿主状态正确。 |
| 进入 `NoteEditorView` 再返回 | 异常 | 回弹提前终止，直接跳回顶部。 |
| 进入 `ReadCalendarView` 再返回 | 异常 | 与书摘编辑页同症状。 |
| 在读首页与时间线都验证 | 都异常 | 说明不是单页私有 `ScrollView`。 |
| 切到搜索 Tab 再回来 | 无法复位 | 说明不是整个 `TabView` 级搜索 UI 一次性重置。 |
| 在阅读栈内打开 `BookSearchView` 再返回 | 可复位 | 关键转折证据。 |

## 3. 关键证据链

### 3.1 受害者是阅读根页，不是单个页面
- `ReadingDashboardView` 的根内容是普通 `ScrollView`：`xmnote/Views/Reading/ReadingDashboardView.swift:78-125`。
- `ReadingTimelineView` 的正式内容根也是普通 `ScrollView`：`xmnote/Views/Reading/Timeline/ReadingTimelineView.swift:83-97`。
- 两个根页都由 `ReadingContainerView` 通过 `KeepAliveSwitcherHost` 挂在同一条阅读栈里：`xmnote/Views/Reading/ReadingContainerView.swift:59-115`。

### 3.2 触发页与复位页共享同一条 `readingPath`
- `NoteEditorView` 作为阅读栈内 push 详情页存在自定义返回拦截：`xmnote/Views/Note/NoteEditorView.swift:321-329`。
- `ReadCalendarView` 也是阅读栈内 push 页，并显式改写导航栏背景状态：`xmnote/Views/Reading/ReadCalendar/ReadCalendarView.swift:81-84`。
- `BookSearchView` 同样从阅读栈通过 `BookRoute.add` 被 push 进去：`xmnote/Views/MainTabView.swift:239-240`。

### 3.3 `BookSearchView` 是同栈里唯一明确重建搜索宿主的页面
- `BookSearchView` 在页面自身挂了 `.searchable(... placement: .navigationBarDrawer(displayMode: .always))`：`xmnote/Views/Book/BookSearchView.swift:198-204`。
- 修复后的 `MainTabView` 只在搜索 Tab 激活时才挂根级搜索宿主：`xmnote/Views/MainTabView.swift:185-186`、`xmnote/Views/MainTabView.swift:428-458`。
- 这说明此前阅读栈长期背着根级搜索宿主，`BookSearchView` 的页面级搜索宿主只是偶然把共享系统状态重绑回了正确值。

## 4. 已证伪假设
- 假设一：`ReadingDashboardView` 自己的 bounce 策略丢了。
  - 证伪：时间线也复现，说明不是首页私有滚动实现。
- 假设二：`ReadingContainerView` 的导航栏隐藏/背景策略污染了根页。
  - 证伪：补导航栏恢复规则后问题仍然存在，而且 `BookSearchView` 不依赖同一套阅读专用导航栏补丁却能复位。
- 假设三：`navigationPopGuard` 是主因。
  - 证伪：`ReadCalendarView` 同样会触发，但它不依赖 `navigationPopGuard`。
- 假设四：`KeepAliveSwitcherHost` 或时间线预热导致页面重建异常。
  - 证伪：首页和时间线都只是受害者；真正能“修复”问题的是同栈搜索页，而不是容器保活或预热。

## 5. 真实 Root Cause
- 修复前，`MainTabView` 在整个 `TabView` 根上长期挂着 `.tabViewSearchActivation(.searchTabSelection)` 和 `.searchable(text: $searchQuery, prompt: ...)`。
- 这意味着当前选中的任意导航栈，包括阅读栈，都可能长期持有根级搜索控制器与其关联的系统滚动宿主状态。
- 当阅读栈中的详情页进行 push/pop，系统宿主状态在某些路径上没有正确回落，导致返回阅读根页后 overscroll 回弹提前终止。
- `BookSearchView` 进入后之所以能复位，是因为它自己的 `.searchable(... navigationBarDrawer(.always))` 在同一条 `UINavigationController` 上强制重新接管并刷新了搜索宿主状态。

## 6. 最终修复
- 保留 `navigationPopGuard` 的 `attachmentGeneration` 硬化，作为独立正确性修复：`xmnote/UIComponents/Foundation/SurfaceComponents.swift:193-231`。
- 把 `MainTabView` 根级搜索宿主改成按搜索 Tab 条件挂载：
  - 搜索 Tab 激活时才挂载 `tabViewSearchActivation(.searchTabSelection)` 与 `.searchable(...)`。
  - 非搜索 Tab 不再长期持有根级搜索宿主。
- 不再把阅读页的 `ScrollView`、bounce、导航栏背景当主修复点。

## 7. 为什么前两轮思路错了
- 第一轮看到的是“回弹异常”，容易自然联想到根页 `ScrollView`、bounce、布局或导航栏背景。
- 但这个问题从一开始就不是单点页面问题，而是共享系统宿主问题；如果不先找共同 owner，就会被表层现象误导。
- 真正改变判断的是两个事实：
  - 首页和时间线同时中招。
  - 只有同栈 `BookSearchView` 能复位，而切到搜索 Tab 不能复位。

## 8. 后续排查 Checklist
1. 同一个 Tab 下多个根页同时出现滚动/回弹异常时，先找共享 `NavigationStack` owner，不要先改页面布局。
2. 如果某个同栈页面进入后能“顺手修好”问题，优先检查它是否挂了系统级 modifier，例如 `searchable`、导航栏背景、搜索抽屉、系统滚动关联能力。
3. 先区分“受害者页面”和“真实写入点”：
   - 受害者页面：出现异常的根页。
   - 写入点：改写共享系统宿主状态的容器或详情页。
4. `KeepAliveSwitcherHost`、scene 恢复、页面预热只有在共享系统 owner 被排除后，才进入下一层排查。
5. 对 iOS 26 新系统能力，优先怀疑 modifier owner 范围，而不是先怀疑业务布局实现。
