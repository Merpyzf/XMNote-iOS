# 搜索交互升级 iOS26 - 设计文档

## 1. 设计概览
- 使用 iOS 26 的 `tabViewSearchActivation(.searchTabSelection)` 作为搜索 Tab 激活策略。
- 搜索输入状态继续由 `MainTabView` 统一管理，但根级搜索宿主只在 `selectedTab == .search` 时挂载。
- `SearchView` 仅负责展示搜索态内容，不再维护输入框状态。

## 2. 状态设计
- `selectedTab`：当前选中主 Tab。
- `searchQuery`：系统搜索输入绑定。
- `searchPath`：搜索 Tab 自己的 `NavigationStack` 路径。
- `mainTabSearchHost`：`MainTabView` 私有搜索宿主封装，负责按搜索 Tab 激活状态条件挂载系统搜索能力。

## 3. 交互流程
1. 用户切到搜索 Tab。
2. `MainTabView` 仅在搜索 Tab 激活时挂载根级 `tabViewSearchActivation(.searchTabSelection)` 与 `.searchable(...)`。
3. 系统搜索输入直接绑定 `searchQuery`，`SearchView` 读取同一份绑定值展示搜索态内容。
4. 用户离开搜索 Tab 后，根级搜索宿主随之卸载，其他导航栈不再继续持有系统搜索控制器状态。

## 4. 视图改造
- `MainTabView`：
  - 搜索 Tab 继续使用现有 `NavigationStack(path: $searchPath)`。
  - 新增 `mainTabSearchHost(isEnabled:searchQuery:)` 私有封装。
  - 只有 `selectedTab == .search` 时才挂载根级搜索宿主。
  - `SearchView` 改为注入 `query` 绑定。
- `SearchView`：
  - 不自带输入框。
  - 只消费 `MainTabView` 提供的 `query` 绑定和结果承载区。
  - 保留 Add 菜单按钮与空态文案。

## 5. 宿主隔离规则
- `searchable`、`tabViewSearchActivation`、`tabBarMinimizeBehavior`、导航栏背景/显示偏好都属于系统宿主状态，不是页面私有布局状态。
- 系统宿主状态必须挂在“最小但正确”的共享 owner 上；能按 Tab 条件隔离时，不能让所有导航栈长期共用同一宿主。
- 当同一条 `NavigationStack` 里既有根页滚动，又有 push 详情页、自定义导航栏或页面内搜索时，优先核对共享容器上的系统 modifier 范围，不要先改根页 `ScrollView`。
- `KeepAliveSwitcherHost` 只负责保活和显隐切换，不承担系统搜索宿主隔离职责。

## 6. 风险与规避
- 风险：根级系统搜索宿主挂载范围过大，导致非搜索 Tab 的导航栈也长期持有搜索控制器状态。
- 风险：多个根页共用同一条导航栈时，某个详情页的 push/pop 可能把系统滚动宿主带入异常状态。
- 规避：按搜索 Tab 条件挂载根级搜索宿主；若同栈出现多个根页同时滚动异常，先排查共享系统 modifier owner，再看页面布局。
