# 首页书架管理分阶段实施方案

## 1. 方案定位

本文承接《执行计划》和《交互设计文档》，用于指导首页书架管理从 Android 向 iOS 的分阶段实现。执行策略采用“纵向切片 + 数据语义闸门”：每个阶段先确认 Android 功能与写入规则，再实现 iOS 端更适合本平台的 UI/UX。

基本原则：

- Android 当前 DAO SQL 是数据修改语义源。
- iOS 可以优化信息架构、视觉层级、编辑态和反馈，但不能改变最终数据结果。
- ViewModel 不直接访问 GRDB；所有本地数据读写必须经过 Repository。
- 每项写操作实现前必须在《执行计划》中完成 `[D]`，验证通过后补 `[V]`。
- 阶段交付优先保证可构建、可验证、可回滚，不跨阶段提前铺开低优先级能力。

## 2. 阶段总览

| 阶段 | 名称 | 交付目标 | 是否写库 |
| --- | --- | --- | --- |
| Phase 0 | 数据语义基线 | 建立 Android 写操作矩阵和 iOS schema 对照 | 否 |
| Phase 1 | 只读书架模型 | Book / Group 混排读取和默认网格展示 | 否 |
| Phase 2 | 书架维度与显示设置 | 默认、状态、标签、来源、评分、作者六维浏览 | 否 |
| Phase 3 | 编辑态与选择体系 | iOS 原生编辑态、混选、底部操作栏 | 仅占位，不接未核对写入 |
| Phase 4 | 排序与置顶 | 手动排序、移到最前/最后、置顶/取消置顶 | 是 |
| Phase 5 | 删除与分组 | 删除书籍、删除分组、移入分组 | 是 |
| Phase 6 | 二级批量管理 | 标签、来源、阅读状态、收藏/书单、导出 | 按动作决定 |
| Phase 7 | 收口与治理 | 文档、对齐情况、学习总结、治理校验 | 否 |

## 3. Phase 0：数据语义基线

### 3.1 目标

在进入任何写入实现前，先把 Android 端首页书架管理的真实 owner、DAO SQL、事务边界和同步风险固定下来。该阶段的价值是防止 iOS 因 UI 优化而误改数据语义。

### 3.2 工作项

- 建立 Android 写操作矩阵：
  - 手动排序
  - 移到最前
  - 移到最后
  - 置顶
  - 取消置顶
  - 删除书籍
  - 删除分组
  - 移入分组
  - 设置标签
  - 设置来源
  - 设置阅读状态
  - 加入收藏/书单
  - 导出书籍/书摘
- 对每项写操作记录：
  - Android UI owner
  - Android Repository owner
  - Android DAO SQL
  - 是否事务
  - `is_deleted` 条件
  - 是否更新 `updated_date`
  - 是否保持 `last_sync_date`
  - 影响表范围
  - iOS Repository 对应接口
  - iOS 到 Android 同步风险
- 核对 iOS schema 与 Record：
  - 确认所有 Android 关联表在 iOS 是否存在。
  - 确认字段名、软删除语义、时间戳单位和默认值。
  - 对缺失或偏差项标 `[!]`，进入决策或修复，不允许直接实现写入。

### 3.3 Android 对齐闸门

- 排序类写入必须确认是否更新 `updated_date`。
- 删除类写入必须逐表确认级联软删除范围。
- 分组类写入必须确认是否先取消置顶、是否更新旧关系、是否插入新关系。
- 批量类写入不得套用通用更新模板，必须逐动作核对。

### 3.4 验收

- 《执行计划》中的数据语义基线任务完成并补充证据。
- 每项写操作至少有一条 Android 代码路径和一条 DAO 证据。
- 所有不确定项明确标 `[!]`。

## 4. Phase 1：只读书架模型

### 4.1 目标

先把 iOS 当前“只读 Book 列表”升级为“Book / Group 混排书架”，只做读取和展示，不写数据库。

### 4.2 关键改动

- Domain 新增书架模型：
  - `BookshelfItem`
  - `BookshelfItemID`
  - `BookshelfBookPayload`
  - `BookshelfGroupPayload`
- `BookshelfItemID` 必须显式区分 Book 与 Group，例如：
  - `.book(Int64)`
  - `.group(Int64)`
- Repository 新增读取接口：
  - `observeBookshelf(setting:searchKeyword:)`
- Repository 读取需要覆盖：
  - Book 基础信息
  - Group 基础信息
  - Group 组内代表封面
  - 笔记数量
  - 置顶状态与 `pin_order`
  - 普通顺序 `book_order` / `group_order`
- ViewModel 升级为书架状态机：
  - loading
  - content
  - empty
  - searching
  - error

### 4.3 UI 实现

- 默认维度先展示 Book / Group 混排网格。
- Book item 使用 `XMBookCover` 渲染封面。
- Group item 使用轻量聚合封面，标题显示组名，副信息显示 `N本`。
- 不引入编辑态、不引入写操作。

### 4.4 Android 对齐闸门

- 默认读取顺序必须对齐 Android `CUSTOM` 语义：
  - 置顶项优先。
  - 置顶项按 `pin_order`。
  - 普通 Book 按 `book_order`。
  - 普通 Group 按 `group_order`。
- 过滤条件必须排除 `is_deleted = 1`。

### 4.5 验收

- iOS 能展示 Book / Group 混排书架。
- 仅改读取和展示，不产生写库。
- 构建命令通过。

## 5. Phase 2：书架维度与显示设置

### 5.1 目标

实现《交互设计文档》定义的六个浏览维度，并把 Android 端偏设置化的能力整理为 iOS 更轻的维度 rail + 设置 Sheet。

### 5.2 关键改动

- Domain 新增：
  - `BookshelfDimension`
  - `BookshelfDisplaySetting`
  - `BookshelfSortMode`
  - `BookshelfSection`
  - `BookshelfAggregateGroup`
- ViewModel 增加：
  - 当前维度
  - 搜索关键词
  - 显示设置
  - 各维度派生数据
- Repository 读取支持维度需要的聚合数据：
  - 状态
  - 标签
  - 来源
  - 评分
  - 作者首字母

### 5.3 UI 实现

- 新增书架维度 rail：
  - 默认
  - 状态
  - 标签
  - 来源
  - 评分
  - 作者
- 各维度布局：
  - 默认：真实书架顺序网格。
  - 状态：状态分区和横向封面条。
  - 标签/来源：两列聚合卡。
  - 评分：纵向评分分组卡。
  - 作者：字母分区 + 右侧索引。
- 首页书架管理内的书籍列表集合区统一采用 `UIViewRepresentable + UICollectionView + UIHostingConfiguration` 架构：
  - SwiftUI 负责页面壳层、维度 rail、Sheet、编辑态栏与导航。
  - UIKit collection view 负责书籍集合区滚动、Grid/List 布局、TabBar 滚动观察与系统底部边缘过渡。
  - SwiftUI cell 继续复用现有书籍/分组卡片视觉。
- 显示设置 Sheet 承载：
  - 网格 / 列表
  - 列数或封面密度
  - 标题显示方式
  - 是否显示书摘数量
  - 排序规则
  - 置顶是否固定

### 5.4 UI 优化边界

- `[优化]` 维度 rail 外露高频浏览方式，减少进入 Android 长设置页的次数。
- `[优化]` 标签、来源、评分、作者采用专属布局，避免同构列表。
- `[对齐]` 所有维度只影响“怎么看”，不改变 Android 数据排序字段。

### 5.5 验收

- 六个维度均可浏览。
- 小屏不出现文字溢出或控件重叠。
- 搜索态保留当前维度上下文。
- 手动排序在非默认维度和搜索态不可用。
- 不写数据库。

## 6. Phase 3：编辑态与选择体系

### 6.1 目标

建立 iOS 原生编辑态，把 Android 较分散的批量管理入口收敛为“顶部选择状态 + 底部操作栏 + 更多菜单”。

### 6.2 关键改动

- ViewModel 新增编辑态状态：
  - `isEditing`
  - `selectedIDs`
  - `isAllSelected`
  - `pendingAction`
  - `lastRecoverableSnapshot`
  - `writeError`
- 支持 Book / Group 混选。
- 支持全选、反选、取消选择。
- 写操作入口先接入状态反馈和禁用逻辑。

### 6.3 UI 实现

- 顶部编辑态：
  - 取消
  - 已选 `N项`
  - 全选 / 反选
- 底部操作栏：
  - 置顶
  - 移动
  - 更多
  - 删除
- 单项 context menu：
  - 添加书摘
  - 置顶 / 取消置顶
  - 编辑书籍
  - 阅读详情
  - 开始计时
  - 删除
- 写操作进行中：
  - 禁用重复点击
  - 展示处理中状态
  - 失败保留编辑上下文

### 6.4 Android 对齐闸门

- 未完成 `[D]` 的写操作只允许展示入口或占位提示，不允许真实落库。
- 批量动作不得绕过 Repository。

### 6.5 UI 优化边界

- `[优化]` Android 批量能力入口较分散，iOS 固定为底部操作栏。
- `[优化]` 长按菜单只承载单项高频动作，批量动作不塞进 context menu。
- `[对齐]` 具体写操作的数据语义仍等待 Phase 4 / 5 / 6 分项核对。

### 6.6 验收

- 编辑态进入、退出、选择、全选、反选稳定。
- Book / Group 混选不会发生 ID 类型混淆。
- 未核对写操作不会落库。

## 7. Phase 4：排序与置顶

### 7.1 目标

完成首页高频管理闭环：手动排序、移到最前、移到最后、置顶、取消置顶。

### 7.2 Android 数据语义

- 手动排序：
  - Android owner：`BookRepository.updateBookDataListOrder`
  - Book 写 `book_order`
  - Group 写 `group_order`
  - 按最终列表下标从 `0` 写入
  - 不更新 `updated_date`
- 移到最前 / 最后：
  - Android owner：`DefaultBookListViewModel.moveBooks`
  - 只移动非置顶项
  - 置顶项保持置顶区
  - 最终仍走 `updateBookDataListOrder`
- 置顶 / 取消置顶：
  - Android owner：`BookRepository.pin` / `cancelPin`
  - Book 与 Group 共用全局最大 `pin_order`
  - 取消置顶写 `pinned = 0, pin_order = 0`
  - 不更新 `updated_date`

### 7.3 关键改动

- Repository 新增并实现：
  - `updateBookshelfOrder(_:)`
  - `pinBookshelfItems(_:)`
  - `unpinBookshelfItem(_:)`
  - `moveBookshelfItemsToStart(_:)`
  - `moveBookshelfItemsToEnd(_:)`
- ViewModel 增加排序提交状态和失败回滚。
- 将默认书架 `UICollectionView` 排序能力生产化，以系统拖拽、让位动画和边缘自动滚动替代 SwiftUI 手写拖拽基建。
- Collection view 只维护本地预览顺序，拖拽结束后把最终 `BookshelfItemID` 顺序交给 ViewModel。

### 7.4 UI 实现

- 手动排序只在以下条件启用：
  - 默认维度
  - 手动排序模式
  - 非搜索态
  - 编辑态
- 拖拽中显示 lifted 状态和目标占位。
- 普通项不得拖入置顶区。
- 置顶项不参与普通排序。
- 松手后一次性提交 Repository。
- 失败后恢复拖拽前顺序。
- 默认维度 Grid/List 共用 `BookshelfDefaultCollectionView`，通过 `UICollectionViewCompositionalLayout` 切换布局。
- `BookGridView` 通过 `isPageActive` 控制默认集合区是否登记为 TabBar observed scroll view，避免隐藏页抢占底部栏滚动联动。

### 7.5 验收

- 《执行计划》对应任务补 `[D]` 和 `[V]`。
- 验证 `book_order` / `group_order` 连续写入。
- 验证置顶追加顺序。
- 验证不额外更新 `updated_date` / `last_sync_date`。
- 验证重新读取后顺序稳定。

## 8. Phase 5：删除与分组

### 8.1 目标

完成高风险写入能力：删除书籍、删除分组、移入分组。该阶段以数据安全优先，UI 优化只服务减少误操作。

### 8.2 Android 数据语义

- 删除书籍：
  - Android owner：`BookRepository.deleteBook`
  - 事务内软删除 book 与所有 book_id 关联数据
  - 逐表核对是否更新 `updated_date`
- 删除分组：
  - Android owner：`BookRepository.deleteBookOrGroup` + `GroupRepository.moveOut` + `GroupRepository.deleteGroup`
  - 先移出组内书籍，再删除分组
  - 移出时取消置顶
  - 软删除 `group_book`
  - 按用户选择写入开头或末尾的新 `book_order`
  - `deleteGroup` 本身不更新 `updated_date`
- 移入分组：
  - Android owner：`GroupRepository.moveBooksToGroup`
  - 事务内取消置顶、删除旧关系、插入新关系、目标分组内追加排序

### 8.3 关键改动

- Repository 新增并实现：
  - `deleteBookshelfItems(_:groupBooksPlacement:)`
  - `moveBooks(_:toGroup:)`
- 建立删除书籍关联表清单，逐表实现软删除。
- 建立分组移动事务，防止半写入。

### 8.4 UI 实现

- 删除书籍使用 `XMSystemAlert`，destructive 语义只用于删除按钮。
- 删除分组必须询问组内书籍去向：
  - 放到书架开头
  - 放到书架末尾
  - 取消
- 移入分组使用 Sheet，支持选择已有分组和新建分组入口。
- 写入中禁用重复提交。

### 8.5 验收

- 《执行计划》对应任务补 `[D]` 和 `[V]`。
- 删除书籍后不存在 Android 会删除而 iOS 遗留的关联数据。
- 删除分组后组内书籍位置与 Android 规则一致。
- 移入分组后置顶状态、旧关系、新关系、组内排序与 Android 一致。

## 9. Phase 6：二级批量管理

### 9.1 目标

补齐 Android 端首页书架的低频管理能力，但全部放入编辑态“更多”或二级 Sheet，避免首页常驻入口膨胀。

### 9.2 工作项

- 批量设置标签。
- 批量设置来源。
- 批量设置阅读状态。
- 批量加入收藏/书单。
- 批量导出书籍。
- 批量导出书摘。

### 9.3 Android 对齐闸门

- 每项动作独立完成 `[D]`。
- 不得使用未经核对的通用批量更新方法。
- 导出若不写数据库，需明确标注“无落库写入”。
- 若某项动作会影响同步字段，必须核对 Android 是否更新 `updated_date`。

### 9.4 UI 优化边界

- `[优化]` 低频动作进入“更多”，不挤占首页底部栏。
- `[优化]` 标签、来源、阅读状态使用专属 Sheet，减少误选。
- `[对齐]` 具体落库规则逐项跟随 Android。

### 9.5 验收

- 每项批量动作均有 `[D]` / `[V]`。
- 首页编辑态仍只保留高频底部操作。
- 批量动作失败时不破坏已选集合和编辑上下文。

## 11. 接口演进建议

### 11.1 Domain

建议新增：

```swift
enum BookshelfItemID: Hashable {
    case book(Int64)
    case group(Int64)
}

enum BookshelfDimension: Hashable {
    case `default`
    case status
    case tag
    case source
    case rating
    case author
}

enum GroupBooksPlacement: Hashable {
    case start
    case end
}
```

同时新增：

- `BookshelfItem`
- `BookshelfOrderItem`
- `BookshelfDisplaySetting`
- `BookshelfSection`
- `BookshelfAggregateGroup`

### 11.2 Repository

建议在 `BookRepositoryProtocol` 中新增：

```swift
func observeBookshelf(
    setting: BookshelfDisplaySetting,
    searchKeyword: String?
) -> AsyncThrowingStream<[BookshelfItem], Error>

func updateBookshelfOrder(_ orderedItems: [BookshelfOrderItem]) async throws
func pinBookshelfItems(_ items: [BookshelfItemID]) async throws
func unpinBookshelfItem(_ item: BookshelfItemID) async throws
func moveBookshelfItemsToStart(_ items: [BookshelfItemID]) async throws
func moveBookshelfItemsToEnd(_ items: [BookshelfItemID]) async throws
func deleteBookshelfItems(
    _ items: [BookshelfItemID],
    groupBooksPlacement: GroupBooksPlacement
) async throws
func moveBooks(_ bookIDs: [Int64], toGroup targetGroupID: Int64) async throws
```

约束：

- Repository 只接收业务意图和最终顺序。
- Repository 内部负责 GRDB 事务与 SQL。
- ViewModel 不拼 SQL，不直接读取 `AppDatabase`。

### 11.3 ViewModel

`BookViewModel` 演进方向：

- 书架订阅。
- 维度切换。
- 搜索上下文。
- 显示设置。
- 编辑态。
- 选中集合。
- 写操作提交态。
- 失败回滚态。

### 11.4 View

- `BookContainerView` 保持页面壳层职责。
- `BookGridView` 保持 SwiftUI 书架内容入口职责，不直接承担书籍集合区滚动和排序实现。
- `BookshelfDefaultCollectionView` 是默认书架集合区基准实现：`UIViewRepresentable` 包装 `UICollectionView`，使用 `UICollectionViewCompositionalLayout` 切换 Grid/List，使用 `UIHostingConfiguration` 承载 SwiftUI cell。
- 后续 `BookshelfBookListView` 等二级书籍列表需要迁移到与默认模块一致的 collection 架构；只读列表不启用拖拽排序，差异仅在 layout、cell 配置和数据源。
- 维度私有子视图放在 `xmnote/Views/Book/Components/`。
- 业务 Sheet 放在 `xmnote/Views/Book/Sheets/`。
- 跨模块复用组件才进入 `xmnote/UIComponents`。

## 12. 阶段验收清单

每阶段至少检查：

- 是否保持 ViewModel 只走 Repository。
- 是否符合 `XMBookCover` 和 `AppTypography` 约束。
- 是否没有引入未核对的数据写入。
- 是否更新《执行计划》的任务标记。
- 是否执行必要构建。

写操作阶段额外检查：

- Android owner 和 DAO SQL 是否已记录。
- `updated_date` / `last_sync_date` 是否与 Android 一致。
- `is_deleted` 条件是否一致。
- 事务边界是否一致。
- Book / Group 混合场景是否覆盖。
- 失败回滚是否可用。

## 13. 推荐实施节奏

- 第一轮：Phase 0 + Phase 1，先完成只读混排书架。
- 第二轮：Phase 2，完成六维浏览和显示设置。
- 第三轮：Phase 3，完成编辑态和选择体系。
- 第四轮：Phase 4，完成排序与置顶。
- 第五轮：Phase 5，完成删除与分组。
- 第六轮：Phase 6，补齐二级批量能力。
- 最后一轮：Phase 7，统一收口治理。

其中 Phase 4 和 Phase 5 是同步风险最高阶段，不允许跳过 `[D]` / `[V]` 标记。
