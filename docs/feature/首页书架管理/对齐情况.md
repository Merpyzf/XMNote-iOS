# 首页书架管理对齐情况

审查日期：2026-05-14

## 审查结论总览

本轮审查对象为 iOS `Book` Tab 首页书架管理，对照 Android 首页书籍/书架模块。审查按真实生产路径追到 View、ViewModel、Repository、DAO/SQL 与写入事务边界，不以旧执行计划作为事实来源。

结论：iOS 首页书架管理的主流程已覆盖 Android 默认书架、七个浏览维度、搜索、显示设置、编辑态、排序、置顶、删除、移组、批量标签、批量阅读状态、作者/出版社重命名与删除等核心能力。UI 表达采用 iOS 原生顶部 chrome、`Menu`、系统弹窗桥接、底部编辑工具栏和 `UICollectionView` 重排，属于合理平台化表达。

上线前仍有 1 个必须修正或确认的写入语义风险：批量设置来源时，Android 只写 `book.source_id`，不更新 `updated_date`，也不额外过滤 `is_deleted/id != 0`；iOS 当前写 `source_id` 的同时更新 `updated_date` 并过滤有效书籍。这会制造 Android 当前实现不会产生的同步时间戳变化，违反本仓库 Android -> iOS 数据语义红线，建议作为 P1 在上线前处理。

本轮只做静态审查与文档落地，未做运行级数据库断言，也未做模拟器视觉验收。运行级验证项在本文风险表中单独标出。

## Android 反向优化建议

| 建议 | 触发点 | 说明 |
| --- | --- | --- |
| 明确来源变更是否应更新 `updated_date` | 批量来源写入差异 | 如果业务认为来源变化应参与增量同步，Android `BookDao.updateBookSource` 应补 `updated_date`、`is_deleted = 0`、`book.id != 0`；如果不应参与同步，iOS 应去掉当前 `updated_date` 写入。两端必须统一。 |
| Android 默认排序写入不应吞异常 | 排序写入可靠性 | Android `BookRepository.updateBookDataListOrder` 当前 catch 后只 `printStackTrace`，调用方难以感知失败；iOS 当前会抛错并回滚/提示。建议 Android 反向优化为返回失败或抛出，避免 UI 误以为排序已保存。 |
| Android 书架搜索态排序约束可显性提示 | 搜索后禁排序 | iOS 已在搜索态提示“搜索结果不支持排序”。Android 可在搜索过滤态隐藏拖拽或显示轻提示，避免用户误解搜索结果能作为完整顺序提交。 |
| 作者/出版社删除确认文案应强调会删除书籍 | 高破坏动作 | iOS 删除作者/出版社弹窗已明确“将删除该维度下书籍”。Android 当前删除确认入口建议同步强化文案，降低误删风险。 |

## 视觉验收建议

上线前建议在 iPhone 17 Pro 模拟器至少覆盖以下场景：

- 默认、状态、标签、来源、评分、作者、出版社七个维度切换。
- 搜索态进入、清空、退出；搜索后排序禁用提示。
- 默认书架编辑态、全选、取消全选、Book/Group 混选。
- 拖拽排序、移到最前、移到最后、置顶、取消置顶。
- 删除纯书籍、删除含分组的混选项，并分别选择组内书籍移到最前/最后。
- 批量标签、批量来源、批量阅读状态，尤其是读完评分与日期。
- 作者/出版社重命名与删除确认。
- 大字体、空书架、仅分组、长书名、无封面、底部栏避让与底部圆角滚动过渡。
