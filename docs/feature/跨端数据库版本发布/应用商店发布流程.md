# 跨端数据库版本发布 - 应用商店发布流程

## 1. 流程目标
- 支持 Android 国内渠道与 iOS App Store 审核时间不一致的现实发布节奏。
- 保证任一端先上线时，本机体验不受影响，跨端数据不会被错误覆盖。
- 在双端均支持新数据库版本后，再恢复跨端备份、恢复和同步能力。

## 2. 角色分工
| 角色 | 职责 |
| --- | --- |
| 产品负责人 | 确认数据库升级是否进入本次版本，决定跨端能力恢复时间 |
| Android 负责人 | 提供 Room schema、迁移说明、国内渠道包和灰度状态 |
| iOS 负责人 | 对齐 Room schema、提供 iOS 兼容包和 App Store 审核状态 |
| 测试负责人 | 验证双端恢复、版本阻断、提示文案和核心功能回归 |
| 发布负责人 | 跟踪渠道审核、灰度覆盖、异常反馈和最终放量 |
| 客服/运营 | 统一用户解释口径，收集版本不一致期间的反馈 |

## 3. 发布阶段

### 阶段 1：数据库升级立项
- 确认新的数据库结构版本号。
- 明确结构变更是否影响备份恢复、同步合并和核心 UI。
- Android 产出 Room schema 与迁移说明。
- iOS 确认对齐方案与兼容验证路径。
- 产品确认本次采用“本机可用，跨端受限”的版本不一致策略。

### 阶段 2：双端兼容包准备
- Android 国内渠道包具备读取和迁移新数据库版本的能力。
- iOS App Store 包具备读取和迁移新数据库版本的能力。
- App 升级后，本机数据库可以正常迁移到新结构；这不是问题，也不需要阻止。
- 新版本生成的备份会携带新的 `user_version`，低版本另一端必须能在恢复前识别并阻断。
- 新字段、新表、新关系不应在另一端未上线前成为跨端用户的必需操作路径。

### 阶段 3：审核与灰度
- iOS 提交 App Store 审核。
- Android 按国内渠道提交审核，可按渠道能力执行灰度。
- Android 国内渠道至少覆盖当前实际发布渠道，例如小米、华为、OPPO、vivo、应用宝等。
- 任一端先上线时，仅承诺本机能力正常，不对所有跨端用户宣称完全恢复。
- 发布负责人每日记录各渠道状态：未提交、审核中、灰度中、已全量、被拒、延期。

### 阶段 4：版本不一致保护
- 若 Android 已上线、iOS 未上线：
  - Android 本机数据库可以升级并正常使用。
  - Android 可以创建本机备份，但该备份会标记为高版本备份。
  - iOS 旧版本可以展示备份信息，但恢复按钮必须阻断或禁用。
  - 面向用户解释为“请升级 iOS 版后恢复”，不能解释为备份损坏。
- 若 iOS 已上线、Android 未上线：
  - iOS 本机数据库可以升级并正常使用。
  - iOS 可以创建本机备份，但该备份会标记为高版本备份。
  - Android 旧版本可以展示备份信息，但恢复按钮必须阻断或禁用。
  - 面向用户解释为“请升级 Android 版后恢复”，不能解释为备份损坏。
- 若任一端发现兼容异常：
  - 暂停跨端恢复入口。
  - 保留本机使用。
  - 发布负责人同步异常范围与预计恢复时间。

### 阶段 5：恢复跨端能力
- 双端均已上线支持新数据库版本。
- 国内 Android 主要渠道达到产品设定覆盖要求。
- App Store 版本已可公开下载。
- 双端备份恢复、跨端打开、核心查询均通过验收。
- 发布负责人确认可以对用户恢复“跨端备份/恢复/同步可用”的产品承诺。

### 阶段 6：异常处理
- 某端审核被拒：
  - 已上线端不回滚本机兼容能力。
  - 未上线端继续阻断高版本备份恢复。
  - 修复被拒原因后重新提交。
- 某国内渠道长期延期：
  - 根据该渠道用户占比决定是否继续等待。
  - 未达到主要渠道覆盖要求前，不对所有跨端用户宣称完全恢复。
  - 渠道内用户仍按“低版本遇高版本备份阻断”的规则保护。
- 发现数据兼容问题：
  - 立即暂停跨端恢复和同步合并。
  - 保留本机数据读写。
  - 修复后重新执行双端恢复验证。

## 4. 发布检查清单
- 新数据库结构版本号已确认。
- Android Room schema 已产出。
- iOS 已完成 schema 对齐。
- 双端都能打开旧版本数据库。
- 双端都能打开新版本数据库。
- 低版本端遇到高版本备份时能安全阻断。
- 阻断文案说明“本机数据不受影响”。
- 阻断文案不使用“数据损坏”“恢复失败”作为主解释。
- Android 国内渠道提交状态已登记。
- iOS App Store 审核状态已登记。
- 放开跨端能力前已完成双端备份恢复验证。

## 5. 发布决策标准
- 可以提交审核：
  - 双端兼容包完成。
  - 低版本保护逻辑已通过验收。
  - 新版本本机数据库迁移正常。
- 可以灰度：
  - 本机核心功能正常。
  - 恢复高版本备份时能安全提示。
  - 未发现数据库打开失败或数据覆盖异常。
- 可以全量：
  - 灰度无严重数据问题。
  - 客服与反馈渠道未出现集中恢复失败。
  - 发布负责人确认渠道状态稳定。
- 可以恢复跨端能力：
  - Android 与 iOS 都已支持新数据库版本。
  - 主要 Android 国内渠道完成覆盖。
  - 双端互相恢复验证通过。

## 6. 客服与用户解释口径
- 如果用户问“为什么不能恢复”：
  - `这份备份来自较新版本，当前 App 暂时无法安全识别其中的数据库结构。请先升级 App，升级后即可继续恢复。`
- 如果用户担心“本机数据会不会丢”：
  - `不会。当前设备上的数据仍可正常使用。系统只是暂时阻止把较新版本备份覆盖到当前设备，避免数据不兼容。`
- 如果用户问“为什么另一端可以用，这一端不能用”：
  - `两端应用商店审核时间不同，另一端可能已经支持新的数据库结构。当前端升级后即可恢复跨端使用。`
- 如果用户问“是不是备份坏了”：
  - `不是备份损坏，而是备份版本较新。升级当前 App 后再恢复即可。`

## 7. 不允许的发布行为
- 不允许低版本 App 静默恢复高版本备份。
- 不允许用自动降级掩盖数据结构不兼容。
- 不允许把数据库版本不兼容提示成“数据损坏”。
- 不允许在主要渠道未覆盖前，对所有跨端用户宣称新版本跨端能力完全恢复。
- 不允许在版本不兼容时执行会覆盖本机数据库的跨端操作。
