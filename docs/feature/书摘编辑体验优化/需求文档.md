# 书摘编辑体验优化 — 需求文档

## 1. 背景

本轮迭代聚焦 iOS 端书摘编辑页，问题集中在两类：

1. 聚焦摘录模式下，首次进入时摘录输入区高度不稳定，初始视觉不符合“先写摘录”的业务意图。
2. 元数据录入区（位置、章节、标签、创建时间）经历多轮调整后，存在实现重复和局部补丁，后续继续迭代风险高。

同时，用户明确提出以下视觉与交互约束：

- 不增加元数据区背景；
- 章节/标签/创建时间右侧箭头与右边缘间距一致；
- 标签空态“添加标签”右对齐，且与箭头保持一致间距；
- 保存按钮使用品牌色背景 + 白色文字；
- 字体层级要收敛，避免“看起来混乱”。

## 2. 目标

1. 聚焦摘录模式在首次进入时即给出稳定且符合预期的高度分配。
2. 元数据区视觉节奏统一，交互 affordance 一致，避免行级差异。
3. 将“补丁叠补丁”实现收敛为可维护结构，降低后续改动成本。
4. 与 Android 业务意图保持一致：优先摘录输入、元数据为次级编辑动作、创建时间默认随系统时间走。

## 3. 范围

### 3.1 本次范围内

- `NoteEditorView` 聚焦模式高度计算收敛；
- 元数据行（章节/标签/创建时间）布局实现收敛；
- 元数据行间距、右侧箭头槽位、空态对齐规则统一；
- 书摘编辑页字体层级入口统一到 `AppTypography`；
- 创建时间自动走时逻辑保持并确认行为。

### 3.2 本次范围外

- 富文本格式能力扩展（新格式、新工具栏动作）；
- 标签体系、章节体系的数据模型变更；
- 数据库存储结构、迁移策略调整；
- Android 端 UI 改动（仅给出反向建议，不直接落地）。

## 4. 验收标准

1. 聚焦摘录模式首次进入时，摘录输入区高度即稳定；输入前后不出现“先矮后跳”。
2. 元数据区三行（章节/标签/创建时间）标题、值区、右附件对齐规则一致。
3. 标签空态文案右对齐且贴近右侧附件槽位（12pt 槽位方案）。
4. 代码层面不再存在元数据三行重复实现；高度计算入口单一。
5. 构建校验通过：`xcodebuild -project XMNote.xcodeproj -scheme xmnote -destination 'platform=iOS Simulator,name=iPhone 17 Pro' build`。
