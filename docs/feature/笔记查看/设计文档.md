# 笔记查看 - 设计文档

## 1. 设计目标
- 用一套稳定的自建 paging 架构承接书摘查看与混合内容查看，规避 `TabView` 在沉浸场景下的布局缺陷。
- 把底部沉浸渐变、安全区延展、ornament 图标热区抽成可复用组件，供书摘、书评、相关内容查看统一使用。
- 在保持 Repository 边界不破坏的前提下，把混合内容 feed、详情读取、最小编辑与硬删除统一收敛到 `ContentRepository`。

## 2. 分层架构

### 2.1 Navigation
- `ContentRoute`
  - 新增 `contentViewer(source:initialItemID:)`，承接混合内容查看入口。
- `MainTabView`
  - 统一根据 `ContentViewerSourceContext + ContentViewerItemID` 分流：
    - `bookNotes` 和 `timeline(.note)` 进入 `NoteViewerView`
    - 其余时间线来源进入 `ContentViewerView`

### 2.2 Data / Domain
- `ContentRepositoryProtocol`
  - 统一提供 viewer feed 观察、详情读取、书评草稿、相关草稿与硬删除事务。
- `ContentRepository`
  - 统一构建三类内容的时间线 feed。
  - 删除统一走主记录 + 子记录硬删除事务。

### 2.3 ViewModel
- `NoteViewerViewModel`
  - 专注 note-only feed。
  - 提供 `visibleNoteItems(radius:)` 和 `prefetchDetails(around:radius:)`，承接窗口化挂载与邻页预取。
- `ContentViewerViewModel`
  - 承接书摘/书评/相关混合 feed。
  - 提供 `visibleItems(radius:)` 和 `prefetchDetails(around:radius:)`，与阅读日历/书摘查看保持同构。

### 2.4 View / UIComponents
- `NoteViewerContentView`、`ContentViewerContentView`
  - 都采用 `ScrollView(.horizontal) + LazyHStack + .scrollTargetBehavior(.paging) + .scrollPosition`。
  - 页内正文继续使用纵向 `ScrollView`，并通过 `contentMargins` + `readableInset` 处理底部可读留白。
- `ContentViewerDetailBodies`
  - 抽离三类正文 body，避免书摘 viewer 与混合 viewer 各自复制正文布局。
- `ImmersiveBottomChrome`
  - 统一底部渐变、ornament 承载、图标热区与安全区覆盖。

## 3. 关键交互流

### 3.1 进入查看器
1. 时间线或书籍详情生成 `ContentViewerSourceContext`。
2. `MainTabView` 通过 `contentRoute(for:initialItem:)` 判断进入 `NoteViewerView` 还是 `ContentViewerView`。
3. Viewer 初始化对应 ViewModel 并开始 observation。

### 3.2 横向滑动
1. ViewModel 提供当前选中项附近的窗口化页面数组。
2. `ScrollView(.horizontal)` 只挂载可见窗口内页面。
3. `scrollPosition` 负责同步页内滚动位置。
4. 手势结束后才提交最终选中页，避免滑动过程中反复回写业务状态。

### 3.3 邻页预取
1. 选中页变化后触发 `prefetchDetails(around:radius:)`。
2. ViewModel 按当前索引向左右预取 1 页详情。
3. 用户继续横滑时，多数情况下能命中详情缓存，降低白屏等待。

### 3.4 底部沉浸 chrome
1. 页面通过 `GeometryReader` 读取底部安全区。
2. `ImmersiveBottomChromeMetrics.make(...)` 根据实测 ornament 高度与安全区生成滚动补偿。
3. `ImmersiveBottomChromeOverlay` 将渐变和悬浮 ornament 一起延展到安全区底部。
4. 页内正文尾部补 `readableInset`，避免最后一段文字被底部操作区压住。

## 4. 关键设计决策
- 不使用 `TabView`
  - 原因：该场景已验证存在内容区域莫名上移的 BUG，且与沉浸式底部 overlay 组合时更难控。
- iOS 允许混合内容连续左右滑动
  - 原因：这是用户明确要求的 iOS 原生增强，能显著降低查看链路中断。
- 删除统一只做硬删除
  - 原因：这是本轮明确业务约束；即使 Android 相关内容当前为软删除，iOS 也不做语义混搭。
- 顶部保持标准导航栏，底部做沉浸
  - 原因：顶部需要维持稳定的标题、分页进度与返回语义；沉浸收益集中在底部正文阅读区。

## 5. 风险与规避
- 风险：窗口化挂载后，`scrollPosition` 指向页不在当前窗口中，可能出现错页。
  - 规避：在 `visibleItemIDs` 变化时同步校正当前 position，并在交互结束后再提交选中态。
- 风险：底部遮罩不覆盖安全区最底部，出现导航手势条区域断层。
  - 规避：`ImmersiveBottomChromeOverlay` 统一 `ignoresSafeArea(edges: .bottom)`。
- 风险：不同 viewer 重复实现正文布局和底部 bar，后续继续分叉。
  - 规避：正文 body 抽到 `ContentViewerDetailBodies`，底部 chrome 抽到 `ImmersiveBottomChrome`。

## 6. 验证策略
- 编译验证：
  - `xcodebuild -project XMNote.xcodeproj -scheme xmnote -destination 'platform=iOS Simulator,name=iPhone 17 Pro' build`
- 人工走查重点：
  - 书摘查看横滑时正文是否稳定。
  - 混合内容 viewer 的左右切换是否覆盖书摘/书评/相关内容。
  - 底部渐变是否完整覆盖到手势导航条区域。
  - 删除当前页后是否正确回退到相邻项或关闭页面。
