# 笔记查看 - 需求文档

## 1. 背景
- Android 端已有成熟的书摘查看能力：从时间线进入后可左右分页、显示进度、底部提供标签/编辑/分享/删除操作。
- Android 端书评与相关内容仍是单页打开，结构上与书摘查看分叉。
- iOS 端这轮目标不是机械翻译 Android 页面，而是在保持业务意图一致的前提下，收敛为一套可扩展的查看架构，并补足更沉浸的底部操作体验。

## 2. 目标
- 书摘查看架构与阅读日历保持一致，弃用 `TabView`，改为自建 horizontal paging。
- 从时间线进入时，书摘、书评、相关内容统一进入同一套全屏查看体系，允许左右滑动查看当前 source 下全部内容。
- 书摘查看、书评查看、相关内容查看共用底部沉浸 ornament 视觉语言。
- iOS 当前删除语义统一只做硬删除。

## 3. 范围
### 3.1 范围内
- 时间线与书籍详情进入查看器的路由重组。
- 书摘查看页的横向滑动、分页进度、懒挂载与邻页预取。
- 书评/相关内容全屏查看器与最小编辑入口。
- 底部沉浸渐变、悬浮 ornament、安全区覆盖与滚动补偿。

### 3.2 范围外
- 不改 Android 现有页面结构。
- 不新增书评/相关内容的 Android 式独立分页查看器。
- 不引入软删除回收站语义。
- 不新增自动化测试要求，本轮以编译通过为验收基线。

## 4. 交互需求
### 4.1 横向切换
- 书摘查看必须支持左右滑动切换当前来源下的全部书摘。
- 时间线进入混合内容查看时，书摘/书评/相关内容可连续左右滑动切换。
- 横向切换后的正文区不得出现 `TabView` 导致的内容上移问题。

### 4.2 页面结构
- 顶部保持标准导航栏，不做沉浸化。
- 底部操作栏使用悬浮 ornament 风格，不采用托管式底栏。
- 底部渐变必须完整覆盖到底部安全区和手势导航条区域。

### 4.3 删除语义
- iOS 端当前仅支持硬删除。
- 删除成功后，若当前 source 仍有相邻项，应自动回退到相邻内容；若 source 已空，则退出查看器。

## 5. 展示规则
- 书摘查看继续保留书名、分页进度、正文、想法、标签与分享操作。
- 书评查看提供编辑、复制、删除。
- 相关内容查看提供编辑、打开链接、复制、删除。
- 三类内容正文排版与底部沉浸 chrome 视觉应保持统一节奏。

## 6. 验收标准
- 书摘查看不再使用 `TabView`。
- 书摘查看与阅读日历的横向切换架构一致，支持窗口化挂载与邻页预取。
- 时间线进入后，书摘/书评/相关内容可在统一 viewer 中左右滑动。
- 底部渐变完整覆盖安全区最底部，不出现裸露底色断层。
- 删除、编辑、复制、分享/打开链接等底部动作保持可用。
- 编译通过：`xcodebuild -project XMNote.xcodeproj -scheme xmnote -destination 'platform=iOS Simulator,name=iPhone 17 Pro' build`。

## 7. 风险与约束
- `TabView` 在该场景存在内容区域异常上移风险，因此本轮禁止继续沿用。
- 混合 feed viewer 会增加单页类型分支，但可以显著降低三类内容查看器的架构分叉成本。
- iOS 当前只做硬删除，和 Android 相关内容软删除存在有意差异，后续若要回归双端一致，需要连同数据库语义一起重审。
