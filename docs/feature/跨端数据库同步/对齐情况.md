# 跨端数据库同步 - 对齐情况

## 1. 结论
当前 iOS Database 已从“独立手写 schema”收敛为“Android Room v40 canonical schema”。双端对齐重点从业务查询层推进到物理数据库文件层：iOS 新库按 Android Room schema JSON 创建，Android v40 备份恢复到 iOS 前经过 staging 校验与安全整理，iOS 导出的库保留 Room 可识别的 `user_version`、`room_master_table`、实体表、索引、外键与 nullable 语义。

当前对齐状态：整库备份/恢复互通已进入可验证状态；记录级实时同步冲突合并未进入本阶段。

## 4. 剩余边界
- 尚未实现记录级同步冲突合并；后续需要单独定义服务端时间、去重 key、冲突策略和 tombstone 传播规则。
- 当前只承诺 Android Room v40 备份与 iOS canonical v40 互通；未来 v41 需要按版本发布流程重新生成 Room schema JSON 并双端同步。
- 不兼容旧 iOS 开发期 schema 或半初始化库；产品尚未正式上线，因此这是有意收敛，不属于用户数据风险。

## 5. Android 端反向优化建议
- Android 端后续每次修改 Room Entity 或 migration 时，必须提交对应版本 schema JSON，避免 iOS 只能依赖生成代码或运行时反查。
- Android 恢复 iOS 备份时，建议与 iOS 一样先进入 staging 校验，再替换正式库，失败时保留本机数据。
- Android 历史备份若继续存在外键缺口，建议在 Android 恢复链路也补充 `PRAGMA foreign_key_check` 诊断，避免跨端问题只在 iOS 暴露。
