# 跨端数据库同步 - 需求文档

## 1. 背景
XMNote 同时维护 Android 与 iOS 客户端。数据库承载书籍、书摘、书评、阅读时长、分类、标签、备份配置等核心数据，是跨端体验一致性的基础。

当前产品目标不是让两端各自维护一套“看起来相似”的 SQLite 表，而是让同一份备份数据库在 Android Room 与 iOS GRDB 上都能被框架层安全识别、打开和继续使用。为此，Android Room v40 schema JSON 被定义为唯一物理 schema 合同，iOS 新库和恢复库都必须对齐该合同。

## 2. 产品目标
- 用户可以把 Android v40 备份恢复到 iOS，恢复后核心页面可正常打开。
- 用户可以把 iOS 导出的备份恢复到 Android，Android Room 不因 schema validation 或 identity hash 失败而拒绝打开。
- 任一端遇到高版本数据库备份时，必须阻断跨端恢复并提示升级，本机数据继续可用。
- 恢复失败时不得覆盖当前设备本机数据库。
- 结构不兼容、外键不可整理或关键数据无法解码时，用户看到的是友好的“暂时无法安全恢复”提示，而不是“数据损坏”。

## 3. 范围
本阶段覆盖整库备份/恢复互通，不覆盖记录级实时同步冲突合并。

覆盖内容：
- Android Room v40 物理 schema 合同。
- iOS 新库 canonical 建表与默认 seed。
- Android 备份恢复到 iOS 的 staging 校验与安全整理。
- iOS 备份恢复到 Android 的 Room 可识别性。
- 低版本 App 遇到高版本备份的保护规则。
- nullable 字段、软删除、默认占位数据和 tombstone 父行的数据语义。

不覆盖内容：
- v41 或更高版本数据库升级。
- 旧 iOS 开发期 schema 自动迁移。
- 记录级同步冲突合并。
- 自动降级备份。
- Android Room 代码架构改造。

## 4. 用户场景
### 4.1 Android 备份恢复到 iOS
用户从 Android 导出或云端下载 v40 备份，在 iOS 端选择恢复。

期望：
- iOS 先检查备份版本和 Room 物理结构。
- 若备份来自当前 iOS 支持的数据库版本，进入 staging 整理。
- 若存在 Android 历史备份中的外键缺口，iOS 在 staging 副本中补齐安全 tombstone 或软删除异常活跃子记录。
- 全部校验通过后才替换正式库。

### 4.2 iOS 备份恢复到 Android
用户从 iOS 导出备份，再在 Android 端恢复。

期望：
- iOS 导出的数据库保持 Room v40 表结构、索引、外键、nullable、`room_master_table` 和 `user_version`。
- Android Room 可以识别 iOS 备份，不因额外 `grdb_migrations` 内部表失败。
- Android 遇到高版本或 schema 不一致备份时阻断恢复，保留本机数据。

### 4.3 版本不一致
某一端先上线了新数据库版本，另一端仍停留在旧版本。

期望：
- 当前设备本机读写不被锁死。
- 低版本端遇到高版本备份时阻断恢复。
- 用户看到“备份来自较新版本，请升级当前 App 后再恢复”。
- 不静默降级，不尝试用低版本 App 打开高版本库。

## 5. 功能需求
| 编号 | 需求 | 验收标准 |
| --- | --- | --- |
| R1 | Android Room schema JSON 作为唯一物理合同 | iOS 新库按 JSON 创建 35 张 Room 实体表、索引、外键和 `room_master_table`。 |
| R2 | iOS 新库可被 Android Room 打开 | `user_version=40`，identity hash 为 `104ccc4da4aae1203a5850535d772d84`，Android schema validation 通过。 |
| R3 | Android v40 备份可恢复到 iOS | staging 通过版本、schema、外键闭包和核心 Record 解码校验后才替换正式库。 |
| R4 | 恢复失败不覆盖本机库 | 任一校验失败时，正式库保持原状，恢复流程返回友好错误。 |
| R5 | 外键历史缺口可安全整理 | 缺失父行生成软删除 tombstone，异常活跃子记录软删除，不硬删用户数据。 |
| R6 | nullable 字段保持 Room 语义 | Android nullable 文本列保持物理 nullable，iOS 在 Record/mapper/UI 层兜底展示。 |
| R7 | 默认占位数据稳定 | `user.id=1`、`source.id=0`、`read_status`、`book.id=0`、`chapter.id=0` 双端一致。 |
| R8 | 高版本备份阻断 | 低版本 App 不恢复高版本库，并提示升级。 |

## 6. 数据语义需求
- `is_deleted` 是跨端软删除事实字段，业务查询必须过滤软删除记录。
- tombstone 父行只用于外键闭包，不应进入 UI 列表、统计或搜索结果。
- `created_date`、`updated_date` 使用毫秒时间戳。
- `last_sync_date` 当前不承担完整记录级冲突解决语义。
- 默认书与默认章节是系统占位数据，不代表用户真实书籍或章节。

## 7. 用户提示需求
| 场景 | 用户文案方向 |
| --- | --- |
| 高版本备份 | 备份来自较新版本，请升级当前 App 后再恢复。 |
| schema 不一致 | 备份结构暂时无法安全识别，本机数据未被覆盖。 |
| 外键不可整理 | 备份内存在无法安全整理的历史关系，本机数据未被覆盖。 |
| 解码失败 | 备份数据暂时无法安全读取，本机数据未被覆盖。 |
| 替换失败 | 恢复未完成，本机数据未被覆盖。 |

文案避免默认使用“数据损坏”“恢复失败导致丢失”等劝退表达。

## 8. 发布与运维需求
- Android 修改 Room Entity 或 migration 时，必须同步提升数据库版本并导出新 schema JSON。
- iOS 以 Android 新 schema JSON 更新 canonical schema 合同。
- 两端审核上线不同步时，跨端恢复遵循版本发布文档中的“本机可用、跨端受限”策略。
- 发布负责人必须确认双端均支持同一数据库版本后，才对用户宣称跨端恢复能力完全恢复。

## 9. 验收矩阵
| 场景 | 验证内容 |
| --- | --- |
| iOS 首次建库 | 35 张实体表、索引、外键、`room_master_table`、`user_version`、默认 seed。 |
| Android 真实备份恢复到 iOS | 物理 schema 通过，staging 整理后 `PRAGMA foreign_key_check` 为空，核心页面可打开。 |
| iOS 备份恢复到 Android | Android Room 可以打开，不报 identity hash 或 schema validation。 |
| 高版本备份 | 低版本 App 阻断恢复并提示升级。 |
| schema 差异备份 | 阻断恢复，错误能定位到表、字段、索引或外键。 |
| 历史外键缺口 | tombstone 与软删除策略生效，不硬删用户数据。 |
