# 书籍选择组件 - 需求文档

## 1. 功能定位

书籍选择组件不是“书摘编辑页里的一个选书弹层”，而是 Book 模块面向全项目复用的统一“书籍选择流”基础能力。

该组件负责帮助用户完成“选哪一本书”的决策闭环，而不是仅展示一个书籍列表。它需要在一次连续任务中完成以下动作：

- 浏览本地书库
- 搜索目标书籍
- 在本地与在线结果之间切换
- 在在线模式下切换具体搜索源
- 在找不到目标书时手动创建书籍
- 将最终结果稳定回填给调用业务页

首版组件需要覆盖以下复用场景：

- 书摘编辑
- 阅读记录 / 阅读计划
- 统计筛选
- 相关书籍绑定
- 其它涉及“从书库或搜索结果中选择书籍”的业务

组件设计目标是沉淀一套可配置、可扩展、可复用的选书能力，避免各业务页面继续手写私有 sheet、私有搜索页或私有回填逻辑。

---

## 2. 背景与问题

### 2.1 当前 iOS 现状

当前 iOS 端仅在书摘编辑场景中存在一个私有本地选书 sheet，其能力上限明显偏低：

- 仅支持本地书籍列表
- 仅支持通过书名 / 作者做简单搜索
- 单击后直接回填当前书摘
- 不支持在线来源
- 不支持多选
- 不支持手动创建书籍
- 不具备统一的配置模型和结果回流语义

这意味着一旦用户“当前书库里没有这本书”，任务会直接中断，用户必须离开当前流转去其它页面补数据，再回到原任务，操作割裂。

### 2.2 当前 Android 现状

Android 端已经有 builder 驱动的 `BookSearchSheetFragment`，其能力更完整，支持：

- 来源范围：`local / online / both`
- 选择模式：`single / multiple`
- 手动创建书籍
- 默认关键字
- 默认已选项
- 多业务场景复用

Android 已经证明“书籍选择”是稳定的业务组件，而不是某一个页面的局部交互。

### 2.3 当前问题总结

| 问题 | 当前表现 | 对用户的影响 |
|------|----------|--------------|
| 任务闭环不完整 | 找不到目标书时无法原地补齐 | 选书任务中断，操作成本高 |
| 组件边界过低 | iOS 仅存在 NoteEditor 私有 sheet | 无法被阅读、筛选、绑定等场景复用 |
| 能力弱于 Android | 无双源、无多选、无手动创建 | 双端感知不一致，迁移质量不足 |
| 回流语义不统一 | 各页面未来容易各自维护返回结果 | 后续扩展成本高，容易出现行为漂移 |

## 3. 目标

### 3.1 产品目标

书籍选择组件需要对齐 Android 的业务能力上限，同时以 iOS 原生方式提供更顺滑的任务流。

### 3.2 首版必须支持的能力

| 能力项 | 要求 |
|--------|------|
| 来源范围 | 支持 `local / online / both` |
| 在线搜索源选择 | 支持在在线模式下选择具体搜索源，并允许按业务配置可用源集合 |
| 选择模式 | 支持 `single / multiple` |
| 手动创建书籍 | 支持，并能在同一选择流内完成 |
| 默认关键字 | 支持由调用方注入 |
| 已选项回显 | 支持单选与多选两类默认已选 |
| 创建成功回流 | 支持创建后稳定回到当前选择流并完成选书 |

### 3.3 体验目标

- 单选模式下，用户点击结果即可完成，不出现额外确认步骤
- 多选模式下，用户能够明确感知当前已选数量与确认动作
- 搜索无结果时，界面必须给出明确下一步动作，而不是停留在空白状态
- 在线搜索时，用户必须能明确知道“当前正在搜哪个平台”，并且可以低成本切换搜索源
- 手动创建书籍必须是一级能力，不能藏在二级菜单或“更多”入口里
- 创建流程应尽量留在同一任务上下文中完成，避免跳出再返回带来的割裂感
- 同一套组件配置能力要能被多个业务场景复用，而不是继续为每个场景复制一套变体

---

## 4. 非目标

本 PRD 不包含以下内容：

- 书籍录入页自身的完整交互重做
- 在线搜索源站点、抓取能力或排序策略升级
- 推荐词、联想词、个性化排序与高级筛选
- 数据库 schema、Repository、网络接口与缓存实现细节
- 具体 SwiftUI/UIKit 代码实现方案

本需求聚焦“统一的书籍选择流”，不下沉到技术实现层。

## 5. 用户与典型场景

### 5.1 核心用户

- 正在编辑书摘、需要为内容绑定书籍的用户
- 需要从现有书库中选择一本或多本书进行关联的用户
- 在本地找不到目标书、希望立即补齐数据再继续任务的用户

### 5.2 典型场景清单

| 场景 | 触发入口 | 用户目标 | 理想完成路径 | 失败或空结果补救路径 |
|------|----------|----------|--------------|----------------------|
| 书摘编辑绑定已存在书籍 | 书摘编辑页的书籍选择入口 | 快速绑定一本本地已有书籍 | 打开书籍选择组件，搜索或浏览本地列表，点选后立即回填 | 若搜不到，直接手动创建书籍 |
| 本地书库无目标书 | 书摘编辑 / 阅读计划 / 绑定场景 | 在当前任务中补齐目标书 | 搜索无结果后，点击“手动创建书籍”，创建成功后自动回填 | 若创建失败，保留当前关键词与上下文，允许重试 |
| 双源切换选书 | 支持 `both` 的业务页 | 在本地与在线结果中选择目标书 | 先搜本地，无结果时切到在线，点选完成 | 在线失败时提供重试或改为手动创建 |
| 在线多源检索 | 在线或双源业务页 | 在不同平台中寻找更完整的目标书信息 | 进入在线结果后切换搜索源，保留关键词并刷新结果 | 若某源失败，可切换其它源或改为手动创建 |
| 多选批量关联 | 统计筛选 / 书单管理 / 相关书籍绑定 | 一次选择多本书 | 勾选多项，底部确认后统一回流 | 可继续搜索、切换来源、取消部分选择 |
| 在线结果转创建再回填 | 在线搜索页内无完全匹配结果 | 以在线结果为参考补齐一本新书 | 进入创建流，完成后回到选择器或直接完成绑定 | 若未完成创建，返回选择器继续搜索 |

### 5.3 场景设计原则

- 用户任务的真正目标是“完成选书”，不是“浏览列表”
- 无结果时必须有补救动作
- 创建书籍不是额外任务，而是选书任务中的自然分支
- 取消操作不应污染调用页当前状态

## 6. 核心能力模型

本组件以产品配置能力建模，而不是以页面样式建模。

### 6.1 配置维度

| 维度 | 可选值 / 说明 |
|------|----------------|
| 来源范围 | 本地、本地+在线、仅在线 |
| 在线搜索源集合 | 可选，默认支持当前项目已有的在线搜索源集合 |
| 默认在线搜索源 | 可选，由调用方或组件默认策略确定 |
| 选择模式 | 单选、多选 |
| 手动创建书籍 | 开启 / 关闭 |
| 默认搜索词 | 可选，由调用方注入 |
| 默认已选书籍 | 单选一项 / 多选集合 |
| 页面标题 | 默认 `选择书籍`，允许按场景配置上下文文案 |
| 上下文提示 | 可选，用于说明当前业务目标 |

### 6.2 行为语义

| 行为 | 规则 |
|------|------|
| 单选结果点击 | 点击后立即完成并回流，不显示底部确认栏 |
| 多选结果点击 | 点击后切换选中状态，底部显示“添加所选书籍” |
| 来源切换 | 不清空用户已输入关键词，保留搜索上下文 |
| 在线搜索源切换 | 仅刷新在线结果，不清空关键词与已选状态；需明确反馈当前源 |
| 手动创建成功 | 单选模式自动完成；多选模式返回列表并加入已选集合 |
| 取消 | 关闭选择流并返回空结果，不隐式改写调用页状态 |

### 6.3 回流结果语义

| 模式 | 回流结果 |
|------|----------|
| 单选 | 返回单本书籍结果 |
| 多选 | 返回书籍集合结果 |
| 取消 | 返回空结果 |
| 创建成功（单选） | 直接返回新建书籍结果 |
| 创建成功（多选） | 返回选择器，用户确认后统一回流 |

## 7. 信息架构与 UI/UX 方案

### 7.1 总体交互原则

- 搜索栏始终位于顶部，支持搜索书名、作者、ISBN
- 空状态必须区分“书库为空”和“搜索无结果”
- 手动创建书籍是一级动作
- 来源切换不应破坏当前关键词与任务上下文
- 在线搜索源选择必须是显式可见控件，不能藏在二级菜单
- “来源范围”和“在线搜索源”必须是两层不同语义，不能混用成同一组切换按钮
- 在线结果行需要展示来源标识，帮助用户理解结果来源
- 组件应保持 iOS 原生导航语义，不在导航栏按钮上叠加额外 glass/material

### 7.2 主交互流

#### A. 本地单选

用户从业务页面进入书籍选择流，默认展示本地书库结果。若用户点选某一本书，则立即完成选择并回填。

适用线稿：

- `09-书籍选择-本地单选-默认态`
- `10-书籍选择-本地单选-搜索结果`

#### B. 本地无结果 / 空书库

当本地没有任何书籍时，界面需要表达“书库为空”；当用户输入关键词但未找到匹配结果时，界面需要表达“没有找到匹配的书”。这两个状态不能混用，因为它们对应的用户认知与下一步动作不同。

适用线稿：

- `11-书籍选择-空书库`
- `12-书籍选择-无结果-手动创建引导`

#### C. 双源切换

当场景支持 `both` 时，界面允许在本地和在线结果之间切换。切换应视为同一选书任务中的不同来源，而不是进入另一个完全独立的页面。

适用线稿：

- `13-书籍选择-双源模式-本地`
- `14-书籍选择-双源模式-在线结果`

#### D. 在线搜索源选择

当用户进入在线结果后，组件需要提供明确的“搜索源选择条”。该控件用于切换具体平台，例如当前项目已有的在线书源集合。它的语义不是“本地/在线”的范围切换，而是“在线结果来自哪个平台”的平台选择。

交互要求：

- 仅在当前配置支持在线结果时显示
- 当来源范围为 `both` 时，仅在用户切到在线结果视图后显示
- 切换平台时保留当前关键词
- 切换平台后自动刷新在线结果
- 若某一平台搜索失败，允许直接切换其它平台继续搜索
- 结果列表行内要显示来源标识，避免用户混淆当前结果来自哪个平台

#### E. 手动创建与创建后回填

当用户无法从现有结果中找到目标书时，组件内部应提供手动创建书籍入口。创建完成后，需要稳定回到当前任务并完成结果回填，而不是让用户自己重新发起选择。

适用线稿：

- `15-书籍选择-手动创建书籍`
- `16-书籍选择-创建成功回填`

### 7.3 默认文案

| 场景 | 文案 |
|------|------|
| 页面标题 | `选择书籍` |
| 搜索提示 | `搜索书名、作者、ISBN` |
| 空书库标题 | `还没有书籍` |
| 空书库说明 | `先创建一本书，后续书摘才能关联到阅读对象。` |
| 无结果标题 | `没有找到匹配的书` |
| 无结果说明 | `你可以继续修改关键词，或直接手动创建。` |
| 创建 CTA | `手动创建书籍` |
| 多选确认 | `添加所选书籍` |

### 7.4 交互细节要求

- 单选模式下，结果项点击就是完成动作，避免二次确认
- 多选模式下，底部确认栏只在有选中项时出现
- 来源切换条仅在配置支持多来源时出现
- 在线搜索源条仅在配置支持在线结果时出现，且与来源范围切换条分层展示
- 在线搜索源条默认使用横向 pills，支持当前项目现有在线源扩展
- 搜索输入应持续保留，避免在切换来源、创建返回后被清空
- 在线搜索源切换后必须立即反馈当前选中源与搜索状态
- 创建成功后，结果回填必须稳定且可预期，不能要求用户重新搜索一遍

## 8. 状态与反馈

### 8.1 状态分支

| 状态 | 用户看到什么 | 允许做什么 | 下一步 |
|------|--------------|------------|--------|
| 首屏加载 | 占位或加载态 | 等待数据完成 | 展示本地结果或空状态 |
| 本地列表结果 | 本地书籍列表 | 搜索、选择、切换来源、创建 | 完成选书或继续筛选 |
| 在线搜索中 | 搜索中反馈 | 修改关键词、取消、等待 | 展示在线结果或错误态 |
| 在线搜索源切换中 | 当前平台切换反馈 | 等待、改关键词、切换到其它平台 | 刷新为新平台结果 |
| 在线搜索失败 | 错误提示与重试入口 | 重试、改关键词、切回本地、手动创建 | 重试或改走其它路径 |
| 空书库 | 空状态卡 + 创建 CTA | 手动创建书籍 | 创建成功后回填 |
| 无结果 | 无结果卡 + 创建 CTA | 修改关键词、切换来源、手动创建 | 得到结果或创建新书 |
| 创建中 | 创建提交中反馈 | 禁止重复提交 | 创建成功或失败 |
| 创建成功回填 | 成功后的结果回流 | 单选自动结束，多选回到列表 | 结束当前任务或继续确认 |

### 8.2 异步反馈原则

- 搜索、在线加载、手动创建、补齐详情等异步动作必须有即时可感知反馈
- 在线搜索源切换不能表现为“无响应”，必须给出加载或骨架反馈
- 不允许“点击后没有任何反馈”
- 创建与搜索进行中，应避免重复触发同一动作
- 失败状态必须给出下一步动作，而不是仅展示错误信息

## 9. 跨平台对齐要求

### 9.1 与 Android 必须对齐的业务能力

| 能力 | 对齐要求 |
|------|----------|
| 来源范围 | 必须支持 `local / online / both` |
| 在线搜索源 | 必须支持与当前项目在线书源能力一致的源选择 |
| 选择模式 | 必须支持 `single / multiple` |
| 手动创建 | 必须支持 |
| 默认关键字 | 必须支持 |
| 默认已选 | 必须支持 |
| 多场景复用 | 必须支持 |

### 9.2 iOS 端优化目标

在业务能力对齐 Android 的前提下，iOS 端应提供更顺滑的交互表达：

- 单选点选即完成，减少多余确认步骤
- 搜索无结果时，CTA 更直接、更靠前
- 创建与回填尽量在同一 `NavigationStack` 内完成，降低任务割裂
- 在线搜索源选择使用显式横向来源条，避免 Android 式二次菜单造成的路径折返
- 减少事件总线式跳转和“跳出去再回来”的体验负担

### 9.3 对齐原则

- 业务能力对齐 Android
- 交互表达采用 iOS 原生体验
- 不机械照搬 Android 的页面结构，而是对齐“用户目标是否能更顺畅完成”

## 10. 需求边界与组件归属

### 10.1 组件归属

书籍选择组件属于 Book 模块的基础业务组件，不是 `UIComponents` 层的纯展示组件。

原因是该组件不仅负责视图渲染，还负责：

- 来源切换与搜索任务编排
- 选中态管理
- 创建流转与回填
- 结果语义统一

因此，纯展示行、状态卡、来源切换条可以沉淀为复用 UI 子组件，但整个“书籍选择器状态机”必须归于业务组件层。

### 10.2 统一扩展原则

- 不允许业务页面继续手写私有书籍选择 sheet
- 后续如有差异场景，应优先通过统一配置扩展组件
- 只有当出现明确、稳定、可复用的新语义时，才允许在统一组件上增加新能力
- 不允许为了某个业务快速上线而复制一套页面变体

## 11. 验收标准

| 验收项 | 标准 |
|--------|------|
| 本地单选可用 | 书摘编辑等场景可通过组件完成本地选书 |
| 无结果可补救 | 搜不到书时可直接手动创建书籍 |
| 创建后回填稳定 | 创建成功后能自动回填当前业务页，不要求用户重新选一遍 |
| 双源切换可用 | `both` 模式下可切换本地 / 在线来源 |
| 在线搜索源可切换 | 在线模式下可切换具体搜索源，且保留关键词与当前任务上下文 |
| 多选行为稳定 | 多选模式具备稳定勾选与确认行为 |
| 单选无冗余步骤 | 单选模式不出现额外确认按钮或多余跳转 |
| 空状态有明确 CTA | 空书库与无结果两类状态都给出明确下一步动作 |
| 组件可复用 | 不依赖 NoteEditor 私有类型，可供其它业务场景接入 |
| 结果语义明确 | 单选、多选、取消、创建成功的回流语义清晰一致 |

## 12. 默认决策与假设
