# 书籍选择组件 - 需求文档

## 1. 功能定位

书籍选择组件是 Book 模块面向全项目复用的统一“选书任务流”，目标不是展示一个列表，而是帮助业务页稳定完成“找到书、选中书、必要时补齐书、把结果回填给调用方”这一整段任务。

当前 iOS 已经把能力边界从“书摘编辑页私有选书 sheet”扩展为统一配置组件，并补上了 Android 现有 builder 方案里最核心的能力维度：

- 本地 / 在线 / 双源选择范围
- 单选 / 多选
- 手动创建书籍
- 默认搜索词
- 默认已选项回显
- 在线结果补齐后回流
- 多业务场景复用配置

## 2. 当前现状

### 2.1 iOS 当前落地状态

本次收口后，iOS 已具备统一 `BookPickerConfiguration` + `BookPickerResult` 语义，核心实现集中在：

- `BookPickerConfiguration`：统一标题、来源范围、选择模式、创建入口、默认上下文和在线源配置
- `BookPickerSelection`：统一本地书与在线直返结果
- `BookPickerViewModel`：统一本地查询、在线搜索、选中态、远端补齐和结果确认
- `BookPickerView`：统一搜索层、来源控制层、结果层和多选确认层
- `BookSelectionTestViewModel` / `BookSelectionTestView`：集中映射 20 个 Android 入口场景并提供结果预览

当前正式业务页仍只让 `NoteEditorView` 消费 `.local(BookPickerBook)`，其余 Android 场景先通过调试中心验证配置与回流语义，不在本轮同步扩散到更多正式页面。

### 2.2 Android 参考现状

Android 端现有 `BookSearchSheetFragment.Builder()`，已在 `NoteEditActivity`、`RelevantListFragment`、`ChapterManagerActivity`、`ReadTimeRecordActivity`、`ReadPlanEditActivity` 等多个页面复用，Builder 维度主要覆盖：

- `bookSearchSource(local / online / both)`
- `supportManualCreate(...)`
- `supportMultipleSelect()`
- `checkedBookId(...)`
- `selectedBookIds(...)`
- `defaultSearchKeyword(...)`
- `forMatchChapter()`
- `what(...)`

这说明“书籍选择”在 Android 已经是稳定业务组件，而不是某个页面私有交互。

## 3. 本次需求收口

### 3.1 必须支持的能力

| 能力项 | 当前要求 |
| --- | --- |
| 来源范围 | 支持 `local / online / both` |
| 在线搜索源 | 支持调用方限制在线源集合，并指定默认在线源 |
| 选择模式 | 支持 `single / multiple` |
| 手动创建 | 支持 inline editor / separate page / nested page 三种创建入口策略 |
| 默认上下文 | 支持默认搜索词和默认已选本地书 |
| 在线结果回流 | 支持“必须先转本地书再回填”与“在线结果直返”两种策略 |
| 多选确认 | 支持“必须选中至少一项”与“允许空集合确认”两种策略 |
| 结果语义 | 单选返回一个 `BookPickerSelection`，多选返回 `[BookPickerSelection]` |
| 场景校验 | 支持用调试中心映射 Android 20 个入口场景做配置验证 |

### 3.2 当前明确采用的两类策略

#### A. 在线结果回流策略

`BookPickerOnlineSelectionPolicy` 分为两类：

- `requireLocalCreation`
  先把在线结果补齐成 `BookEditorSeed`，进入创建链路或编辑链路，再回流本地书。
- `returnRemoteSelection`
  点击在线结果后，直接把 `BookSearchResult + BookEditorSeed` 组成 `BookPickerRemoteSelection` 回给业务页。

#### B. 多选确认策略

`BookPickerMultipleConfirmationPolicy` 分为两类：

- `requiresSelection`
  多选模式下必须至少选中一项才能确认。
- `allowsEmptyResult`
  允许用空集合表达“全部书籍 / 未限制范围”等过滤语义。

## 4. 典型场景

### 4.1 已正式接入的场景

| 场景 | 当前行为 |
| --- | --- |
| 书摘编辑关联书籍 | 仅本地单选，允许创建，创建成功后回填本地书 |

### 4.2 已完成配置验证的 Android 场景分组

| 分组 | 场景数 | 业务意图 |
| --- | --- | --- |
| 本地单选 + 创建 | 5 | 书摘编辑、读书计时、读书计划、继续阅读、悬浮球默认书籍 |
| 本地单选 | 4 | 导入映射、打卡、导出单书、批量移动笔记目标书 |
| 本地多选筛选 | 6 | 小组件、复习、批量导出、纸条范围等过滤语义 |
| 双源直接消费 | 3 | 相关书籍、读书计时相关书籍、编辑收藏集合 |
| 在线专项直接消费 | 2 | 章节同步、补全书籍信息 |

## 5. 回流语义

### 5.1 统一结果模型

| 结果 | 含义 |
| --- | --- |
| `cancelled` | 用户取消，不改写调用页状态 |
| `single(BookPickerSelection)` | 单选完成，结果可以是本地书，也可以是在线直返结果 |
| `multiple([BookPickerSelection])` | 多选完成，结果顺序按用户选择顺序返回 |
| `addFlowRequested` | 当前配置要求跳到独立新增书籍页，由上层继续接管 |

### 5.2 选择项模型

| 选择项 | 含义 |
| --- | --- |
| `.local(BookPickerBook)` | 已有本地书或创建成功后回填的本地书 |
| `.remote(BookPickerRemoteSelection)` | 在线结果直返，携带原始搜索结果和补齐后的编辑种子 |

### 5.3 当前正式业务消费约束

为避免功能面在本轮继续扩散，当前正式业务页只允许：

- `NoteEditorView` 继续消费 `.local(BookPickerBook)`

以下能力先保留在调试中心验证：

- `.remote(BookPickerRemoteSelection)` 正式业务消费
- 本地与在线混合多选的正式业务入口
- 章节同步、补全书籍信息等在线专项页面的正式接线

## 6. 交互要求

### 6.1 通用交互原则

- 搜索栏固定在顶部控制区
- 来源范围和在线搜索源必须分层表达，不能混成一组控件
- 本地/在线切换时必须保留当前关键词
- 在线源切换时必须保留关键词并立即刷新结果
- 无结果和失败态都必须给出下一步动作
- 所有异步动作都必须有可感知反馈

### 6.2 单选与多选

- 单选模式：点击结果即完成，不再额外显示确认栏
- 多选模式：点击结果只切换选中态，底部显示确认栏
- 若 `multipleConfirmationPolicy == .allowsEmptyResult`，即使当前选中数为 0，确认栏也需要出现，并用“未限制”表达空集合语义

### 6.3 创建流与在线补齐

- 手动创建必须保留在当前任务语境内完成
- 在线结果若需补齐详情，界面必须给出阻塞态反馈，避免重复点击
- 多选确认若包含远端结果，必须先按用户选择顺序补齐所有远端载荷，再统一回流

## 7. 非目标

本次需求收口不包含以下内容：

- 在本轮把 Android 的 20 个入口全部正式接入 iOS 页面
- 重写 Android 现有书源抓取或排序逻辑
- 对 `NoteEditorView` 以外的正式页面直接开放 `.remote` 结果消费
- 为所有业务场景新建独立页面或独立组件分支

## 8. 验收标准

| 验收项 | 标准 |
| --- | --- |
| 统一模型完成 | `BookPickerConfiguration`、`BookPickerSelection`、`BookPickerResult` 已收口统一语义 |
| 正式场景可用 | `NoteEditorView` 可稳定完成本地单选与创建回填 |
| 双源能力可运行 | 调试中心可验证本地 / 在线 / 双源配置及在线源切换 |
| 远端直返可运行 | 调试中心可验证 `.remote(BookPickerRemoteSelection)` 回流 |
| 多选空集合可运行 | 过滤类场景可验证“空集合 = 全部 / 未限制” |
| 场景映射可追溯 | 调试中心已登记 20 个 Android 入口场景 |
| 结果顺序稳定 | 混合多选确认按用户选择顺序回流 |

## 9. 默认决策

- 本次把 `BookSelectionTestView` 视为正式交付的一部分，用来承接 Android 场景映射验证。
- 本次先收口统一能力与调试验证，不在一个提交里把所有 Android 入口同步接线到正式页面。
- 正式业务页在下一阶段逐步开放 `.remote(BookPickerRemoteSelection)` 消费，而不是现在一次性放开。
