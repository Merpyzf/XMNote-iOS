# 书籍选择组件 - 设计文档

## 设计目标

- 把 iOS 选书能力从 `NoteEditor` 私有 sheet 提升为 Book 模块统一任务流。
- 对齐 Android `BookSearchSheetFragment.Builder()` 的业务意图，同时使用更适合 iOS 的状态机、导航和回流模型。
- 先收口统一配置、结果语义和调试验证，再逐步把正式业务入口迁入统一组件。

## 1. 组件分层

书籍选择组件采用“单一任务流 + 分层控制区”的结构。

### 1.1 页面结构

从上到下分为五层：

1. 导航层
2. 搜索输入层
3. 来源控制层
4. 结果与状态层
5. 多选确认层

### 1.2 各层职责

| 层级 | 职责 | 不负责什么 |
| --- | --- | --- |
| 导航层 | 标题、返回、取消 | 不承载来源切换或搜索源切换 |
| 搜索输入层 | 输入关键词，持续保留上下文 | 不表达来源范围 |
| 来源控制层 | 切换本地/在线范围，切换在线源 | 不直接展示结果 |
| 结果与状态层 | 列表、空态、错误态、加载态、创建入口 | 不做全局确认 |
| 多选确认层 | 多选确认与已选数量/空集合语义反馈 | 单选模式下不出现 |

## 2. 领域模型设计

### 2.1 配置模型

统一入口为 `BookPickerConfiguration`，当前包含这些关键维度：

| 维度 | 字段 |
| --- | --- |
| 标题与上下文 | `title`、`defaultQuery` |
| 来源范围 | `scope` |
| 选择模式 | `selectionMode` |
| 创建策略 | `allowsCreationFlow`、`creationAction` |
| 在线回流策略 | `onlineSelectionPolicy` |
| 多选确认策略 | `multipleConfirmationPolicy`、`multipleConfirmationTitle` |
| 默认已选 | `preselectedBooks` |
| 在线源配置 | `onlineSources`、`preferredOnlineSource` |

这比 Android Builder 更明确的一点是：iOS 把“在线结果怎么回流”和“多选能不能空确认”提升成了显式策略，而不是留给调用页隐式约定。

### 2.2 结果模型

统一结果为 `BookPickerResult`：

- `cancelled`
- `single(BookPickerSelection)`
- `multiple([BookPickerSelection])`
- `addFlowRequested`

统一选择项为 `BookPickerSelection`：

- `.local(BookPickerBook)`
- `.remote(BookPickerRemoteSelection)`

其中 `BookPickerRemoteSelection` 同时保留：

- 原始 `BookSearchResult`
- 已补齐的 `BookEditorSeed`

这样做的原因是：

- 调用页可以直接消费远端结果，而不必强制先落库
- 若后续要进入编辑页，也不需要再次补齐详情
- 单选和多选都能用同一种返回载荷表达本地与远端混合结果

## 3. 状态机设计

### 3.1 可见范围状态

`BookPickerVisibleScope` 只关注“当前正在看哪类结果”：

- `.local`
- `.online`

它和配置层的 `scope(local / online / both)` 不是同一概念：

- `scope` 决定调用方允许哪些来源
- `visibleScope` 决定当前界面正在展示哪一层结果

### 3.2 结果区状态

`BookPickerStatus` 当前分为：

- `localLoading`
- `localResults`
- `localEmptyLibrary`
- `localNoResults`
- `onlineIdle`
- `onlineLoading`
- `onlineResults`
- `onlineFailure(String)`
- `onlineNoResults`

这样拆分的目的，是把“空书库”“本地无结果”“在线未搜索”“在线搜索失败”这些不同用户认知明确分开，避免 UI 用一个空态混过去。

### 3.3 关键行为

| 行为 | 设计决策 |
| --- | --- |
| `updateQuery` | 本地范围实时刷新；在线范围仅更新关键词，不自动发搜索 |
| `switchVisibleScope` | 保留关键词与已选状态，只切换展示结果 |
| `selectOnlineSource` | 只在在线语义下触发刷新，并清空旧的在线错误/结果 |
| `handleLocalBookTap` | 单选直接完成，多选切换选中态 |
| `handleRemoteResultTap` | 按 `onlineSelectionPolicy` 决定“直接回流远端结果”还是“进入创建/编辑链路” |
| `confirmMultipleSelection` | 按 `selectionOrder` 输出稳定顺序的本地/远端混合集合 |

## 4. 关键交互决策

### 4.1 来源范围和在线源严格分层

“本地 / 在线”与“豆瓣 / 番茄 / 晋江 / 起点”等在线源不是同一语义，因此必须拆成两层：

- 第一层：任务范围切换
- 第二层：在线平台切换

这能避免 Android 旧交互里“底部 sheet + 额外设置入口”带来的路径折返，也避免把所有 chips 堆到一排造成认知混乱。

### 4.2 单选点选即完成，多选走确认栏

- 单选：结果项点击就是完成
- 多选：结果项只切换选中态，由底栏统一确认

额外规则：

- 若 `multipleConfirmationPolicy == .allowsEmptyResult`，底栏在 0 选中时也必须保留
- 0 选中时，数量位使用“未限制”而不是数字 `0`

### 4.3 在线结果允许两条完成路径

在线结果点击后有两种路径：

1. `requireLocalCreation`
   先把远端结果补齐成 `BookEditorSeed`，再进入创建/编辑链路。
2. `returnRemoteSelection`
   直接把远端结果回给调用页。

这让组件既能覆盖“必须先落本地书”的传统场景，也能覆盖“业务只需要一个远端 payload”的专项场景。

### 4.4 远端补齐必须有阻塞反馈

远端结果补齐和多选确认时，可能需要异步抓详情。当前设计要求：

- 单点远端补齐时显示“正在补全书籍信息…”
- 多选混合集合确认时显示“正在整理选中书籍…”
- 阻塞态期间禁用重复确认和重复点击

## 5. 当前落地策略

### 5.1 正式业务页

当前正式页面只接入一条安全路径：

- `NoteEditorView`
  - `scope: .local`
  - `selectionMode: .single`
  - `allowsCreationFlow: true`
  - `creationAction: .nestedSearchPage`
  - 只消费 `.local(BookPickerBook)`

这样做的原因是本轮目标是先把统一能力收口，而不是一口气扩散到所有业务页。

### 5.2 调试验证层

`BookSelectionTestViewModel` / `BookSelectionTestView` 承担两个职责：

- 把 Android 20 个入口场景翻译成 iOS 统一配置
- 让本地单选、本地多选、双源混合、在线直返这四类结果都能在一个地方预览

这相当于为后续正式接线提供“配置样板 + 结果样板”，减少边接业务边猜语义的成本。

## 6. 与 Android 的关系

### 6.1 已对齐的部分

- `local / online / both` 的范围配置
- 默认关键词与默认已选
- 手动创建能力
- 单选 / 多选两类选择模式
- 跨页面复用的统一组件边界

### 6.2 当前明确扩大表达力的部分

- iOS 把在线结果回流策略显式建模为 `BookPickerOnlineSelectionPolicy`
- iOS 把多选空集合确认显式建模为 `BookPickerMultipleConfirmationPolicy`
- iOS 用 `BookPickerSelection` 统一本地书与在线直返结果，而不是只能返回本地书
- iOS 新增 20 场景调试中心，集中验证 Android 各入口映射关系

## 7. 风险与规避

### 7.1 风险：把“能力已收口”误当成“所有业务页都已完成迁移”

后果：

- 文档上看起来已经对齐，实际用户入口还没有全部接线

规避：

- 对齐文档单独记录“正式业务接入范围”
- 新业务入口接线前先在调试中心完成配置验证

### 7.2 风险：远端结果补齐失败导致半成品回流

后果：

- 调用页拿到不完整结果，后续流程继续出错

规避：

- 多选确认必须在所有远端结果补齐成功后才回流
- 任一远端补齐失败时整体确认失败，并把错误贴近结果区展示

### 7.3 风险：空集合确认被误解为 Bug

后果：

- 过滤类场景把“空集合”当成“什么都没选”

规避：

- 只在显式声明 `allowsEmptyResult` 的场景开放该能力
- 底栏数量位使用“未限制”提示当前语义
- 调试中心为这类场景单独展示结果预览

## 8. 后续扩展原则

- 优先把已有 Android 业务入口逐个迁到统一组件，而不是继续新建私有选书页
- 新场景接线前先判断是否只是配置差异；只有确实出现新业务语义时，才允许扩展公共模型
- 在正式业务页开放 `.remote(BookPickerRemoteSelection)` 之前，先确认调用方是否真的能消费远端 payload，而不是为了“看起来更强”提前放开
