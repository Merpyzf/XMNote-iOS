# 图片加载管线迁移（Nuke）- 需求文档

## 1. 背景
- 迁移前 iOS 端图片加载能力分散在页面内，缺少统一的请求头、缓存策略与 GIF 处理规则。
- 阅读日历封面取色、书籍网格封面、书籍详情封面属于同一类“远程封面加载”场景，需要共享一套稳定管线。
- Android 端长期通过 `GlideImageLoader` 统一图片加载与防盗链请求头；iOS 需在业务意图上对齐，并用 iOS 原生方式表达。

## 2. 目标
- 引入 Nuke 作为 iOS 统一图片加载底座，完成应用级 `ImagePipeline` 初始化。
- 建立统一请求构造器（URL 合法性、优先级、超时、缓存策略、防盗链请求头）。
- 提供跨模块复用组件 `XMRemoteImage`，统一处理静态图、GIF 探测与占位降级。
- 提供 `XMCoverImageLoading` 抽象，保障非 UI 场景（如封面取色）与 UI 场景共享同一下载策略。
- 增加 Debug 测试页，覆盖静态图/GIF/失败链路/缓存来源可观测性。

## 3. 范围
### 3.1 范围内
- 新增 `xmnote/Infra/ImagePipeline/` 管线基础设施。
- 新增 `XMRemoteImage` 与 `XMGIFImageView`。
- 接入 `BookGridItemView`、`BookDetailView`、`ReadingDurationRankingChart`。
- `ReadCalendarColorRepository` 改造为基于 `XMCoverImageLoading`。
- 新增 `ImageLoadingTestView`、`ImageLoadingTestViewModel` 并挂载到 DebugCenter。

### 3.2 范围外
- Android 端图片栈改造（本次只做 iOS）。
- 非封面类业务图片批量重构（如富文本内联图片）。
- UI 自动化测试新增（本轮仅要求编译通过）。

## 4. 功能需求
### 4.1 统一请求策略
- 仅接受 `http/https` URL。
- 默认请求超时 12s，资源超时 16s。
- 默认缓存策略 `returnCacheDataElseLoad`。
- 对豆瓣域名自动追加 `Referer`，所有请求追加统一 `User-Agent`。

### 4.2 GIF 处理策略
- URL 后缀为 `.gif` 时走 GIF 播放链路。
- 非 `.gif` 但响应头/数据探测为 GIF 时，自动切换到 GIF 播放链路。
- GIF 加载失败必须降级回静态图或占位，不得空白。

### 4.3 非 UI 取色复用策略
- 阅读日历封面取色下载统一走 `XMCoverImageLoading`，禁止直接在 Repository 内拼接下载实现。
- 取色失败必须回退默认色并缓存，避免重复失败开销。

### 4.4 可观测性
- 提供可视化测试页，至少能观测：
  - URL 合法性失败
  - HTTP 失败
  - 超时失败
  - GIF 成功链路
  - 缓存来源（内存/磁盘/网络）

## 5. 验收标准
- 应用启动时成功设置 `ImagePipeline.shared`。
- 书籍网格、书籍详情、月总结排行封面加载路径均切到 `XMRemoteImage`。
- 阅读日历封面取色仓储不再直接依赖原始下载实现，统一经 `XMCoverImageLoading`。
- GIF 能正常播放，失败可降级。
- DebugCenter 可进入“图片加载测试”页面并执行样例。
- iOS 工程编译通过：`xcodebuild -project XMNote.xcodeproj -scheme xmnote -destination 'platform=iOS Simulator,name=iPhone 17 Pro' build`
