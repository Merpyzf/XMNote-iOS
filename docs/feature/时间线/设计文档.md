# 时间线功能设计文档

## 设计目标
- 在 iOS 端恢复 Android 时间线的核心业务链路。
- 保持“日历选日 + 分类过滤 + 多源聚合列表”三段式结构。
- 在列表滚动场景中优先控制视图重建、富文本测量与桥接层刷新成本。
- 消除首次从“在读”切换到“时间线”时的界面破碎感，避免把首开中间状态暴露给用户。

## 问题背景
- 旧实现中，用户首次从“在读”切到“时间线”时，页面会经历多个可感知中间态：
  - 容器先出现空白或半成品结构。
  - 日历、筛选、列表分别在不同时间点补齐。
  - 数据尚未就绪时，会短暂露出伪空态或布局跳变。
- 这类问题的本质不是“加载慢”，而是“结构在加载过程中持续变化”。即使预热成功率很高，只要失败时页面仍会暴露中间态，视觉问题依然存在。
- 因此本次方案从第一性原理出发，把“首帧结构稳定”定义为正确性目标，把“预热命中”降级为性能优化目标。

## 实际架构

### 1. 页面壳层
- `xmnote/Views/Reading/ReadingContainerView.swift`
  - 负责“在读 / 时间线 / 统计”切换。
  - 通过 `KeepAliveSwitcherHost` 保持二级页面切换状态。
  - 在容器层创建并持有 `TimelineViewModel`。
  - 通过 `SubtabBootstrapCoordinator<ReadingSubTab>` 统一调度 `.timeline` 的首开 warmup。
- `xmnote/Views/Reading/Timeline/ReadingTimelineView.swift`
  - 作为时间线正式入口页。
  - 不再自行创建 `TimelineViewModel`，改为承接容器注入。
  - 当容器尚未注入状态源时，先展示静态壳层。

### 2. 状态层
- `xmnote/ViewModels/Reading/TimelineViewModel.swift`
  - 管理 `sections / selectedDate / selectedCategory / displayedMonthStart / markerCache / bootstrapPhase / isRefreshing`。
  - 首次加载时并行完成：
    - 事件列表加载
    - 当前月份 marker 加载
  - 以“首屏快照”一次性提交首屏：
    - `sections`
    - 当前月份 `markerCache`
  - 进入 `ready` 后，再执行相邻月份 marker 预加载。
  - 通过 `sectionsRevision` 降低列表在滚动过程中的深比较成本。
  - 通过 `markerRevision` 控制日历桥接层的数据依赖更新。

### 3. 数据层
- `xmnote/Domain/Repositories/RepositoryProtocols.swift`
  - 定义 `TimelineRepositoryProtocol`，暴露事件查询和月标记查询两条能力。
- `xmnote/Data/Repositories/TimelineRepository.swift`
  - 使用 GRDB 在读事务中执行 6 路查询。
  - Swift 侧合并、排序并按日分组。
  - 整月聚合日历标记，不受列表时间范围配置影响。
- `xmnote/Data/Repositories/RepositoryContainer.swift`
  - 统一组装 `timelineRepository`，供视图层从环境中获取。

### 4. 视图结构
- 顶部日历面板：`TimelineCalendarPanel`
  - 使用 `HorizonCalendar` 横向分页。
  - 通过子树拆分隔离列表 `sections` 对日历桥接层的刷新影响。
  - 仅把 `selectedDate / selectedCategory / markerRevision` 作为日历依赖令牌传入。
- 首开静态日历壳层：`TimelineBootstrapCalendarPanel`
  - 当 `TimelineViewModel` 尚未就绪或首屏快照未完成时，先展示稳定月份头和静态日历网格。
  - 不显示 spinner，也不展示错误空态。
- 列表容器：`TimelineListContainer`
  - 使用 `LazyVStack(pinnedViews: [.sectionHeaders])` 渲染按日分组内容。
  - `TimelineSectionHeader` 承担粘性日期头。
- 首开静态列表壳层：`TimelineBootstrapListShell / TimelineBootstrapListPlaceholder`
  - 使用静态 section header、连接线和卡片版式占位稳定首屏节奏。
  - 保持与真实列表接近的留白和高度，不做 shimmer、骨架或重 loading。
- 卡片层：`TimelineNoteCard / TimelineReviewCard / TimelineRelevantCard / TimelineRelevantBookCard / TimelineTimingCard / TimelineStatusCard / TimelineCheckInCard`
  - 每类事件单独封装，统一通过 `TimelineEventRow` 分发。

## 预热机制详细设计

### 1. 设计原则
- 预热只负责缩短等待，不负责定义页面结构。
- 预热命中与否都不能改变页面的结构语义：
  - 未命中：展示静态壳层。
  - 命中：更快切入正式内容。
- 预热不能侵入 `KeepAliveSwitcherHost`。保活组件只负责视图存活，不应承担业务初始化策略。

### 2. 触发位置
- 触发入口固定在 `ReadingContainerView`：
  - 容器首次 `.task` 创建 `TimelineViewModel` 后，异步发起一次 `.utility` 级别 warmup。
  - 当用户主动切到 `.timeline` 时，再补发一次 `.userInitiated` 级请求。
- 这两个入口都不会直接重复执行真实加载逻辑，最终统一交给 `SubtabBootstrapCoordinator<ReadingSubTab>` 去重。

### 3. 去重与优先级策略
- `SubtabBootstrapCoordinator` 维护每个 subtab 的 `idle / warming / ready` 三态。
- 当 `.timeline` 已进入 `warming` 或 `ready` 后，后续相同请求会被忽略。
- 设计意图：
  - 容器稳定后尽量提前做准备。
  - 用户真的点进时间线时，不再触发第二份首开任务，也不重复打数据库。

### 4. warmup 执行内容
- warmup 只执行 `TimelineViewModel.loadInitialData()`。
- `loadInitialData()` 首次启动时的工作边界固定为：
  - 拉取当前选中日期和分类下的 `sections`
  - 拉取当前月份的 `markers`
  - 原子提交首屏快照
- warmup 阶段不做的事情：
  - 不提前展开相邻月份 marker 大范围预取
  - 不修改用户可见筛选状态
  - 不直接驱动复杂动画或 loading UI

### 5. warmup 与正式进入页面的关系
- 进入时间线页面时只存在两种用户可见结果：
  - `loadInitialData()` 已完成：直接显示正式内容。
  - `loadInitialData()` 未完成：显示静态壳层，等待 ready 后一次性 reveal。
- 也就是说，“预热成功”只是缩短用户停留在静态壳层的时间，不会改变页面结构或交互路径。

## 状态设计

### 1. TimelineViewModel 状态字段
- `bootstrapPhase`
  - `bootstrapping`：首屏快照尚未完成，只允许展示静态壳层。
  - `ready`：正式内容可见。
- `isRefreshing`
  - 用于区分“首次启动”与“已有内容后的刷新”。
  - 切换日期、切换分类、重新拉数据时只标记刷新，不回退到 bootstrap。
- `hasResolvedInitialSnapshot`
  - 标记首屏快照是否已经真正落地，避免重复走首开路径。

### 2. 状态边界
- 首开状态与刷新状态必须分离，不能用一个 `isLoading` 混合表达。
- 混用单一 loading flag 的问题在于：
  - 首次加载和刷新都会触发相同 UI。
  - 结果是已有内容场景也会回退成空壳或 loading，破坏连续性。
- 因此当前设计明确：
  - `bootstrapping` 只出现一次。
  - `isRefreshing` 可以出现多次，但旧内容必须保留在位。

## 关键实现决策

### 0. 预热不是正确性路径，只是优化项
- 当前方案明确把问题拆成两层：
  - 正确性：即便 warmup 完全未命中，时间线首开也必须稳定。
  - 优化：容器在“在读”稳定后，以低优先级尝试提前完成 `loadInitialData()`。
- 因此 warmup 失败时不会退回空白页或 spinner，用户看到的仍是静态壳层。

### 1. 时间范围策略对齐 Android，但先不暴露设置入口
- iOS 已读取 `@AppStorage("timelineDataShowRange")`。
- 默认值保持 Android 一致的“过去半年”。
- 这保证数据语义对齐，但暂不引入额外设置 UI。

### 2. 容器层负责状态创建与启动调度
- `ReadingContainerView` 而不是 `ReadingTimelineView` 负责创建 `TimelineViewModel`。
- `SubtabBootstrapCoordinator` 负责 `.timeline` 的 warmup 去重：
  - 容器首帧稳定后，以 `.utility` 低优先级尝试 warmup。
  - 用户切到时间线时，再以 `.userInitiated` 请求同一路径。
  - 同一个 subtab 只允许首开任务执行一次。
- `KeepAliveSwitcherHost` 仍只负责保活与显隐，不承载业务预热职责。

### 3. 日历和列表拆观察边界
- `ReadingTimelineContentView` 只做容器拼装。
- `TimelineCalendarPanel` 与列表子树独立观察 `TimelineViewModel` 的不同状态字段。
- 目的是避免列表滚动时把 `HorizonCalendar` 桥接层一起拖进刷新热区。

### 4. 首屏快照必须原子提交
- `TimelineViewModel` 新增：
  - `bootstrapPhase`
  - `isRefreshing`
  - `hasResolvedInitialSnapshot`
- `loadInitialData()` 的职责变为：
  - 并发查询当前时间范围的 `sections`
  - 并发查询当前月份 `markers`
  - 单次提交到 `sections / markerCache / bootstrapPhase`
- 在 `bootstrapPhase == .bootstrapping` 期间，页面只允许显示静态壳层。
- 真实空态仅在 `bootstrapPhase == .ready && sections.isEmpty` 时出现。

### 4.1 为什么必须原子提交
- 如果 `sections` 和 `markerCache` 分开提交，用户会先看到半成品时间线，再看到日历标记补齐，仍然会产生“界面是在拼装”的感受。
- 原子提交的目标不是绝对减少总耗时，而是压缩“用户能看到的结构变化次数”。
- 对首帧体验来说，1 次完整 reveal 明显优于 3 次局部补丁式刷新。

### 5. 粘性头必须依赖 `Section + pinnedViews`
- 时间线日期头不是普通 overlay，而是 `LazyVStack` 的 section header。
- 这样可以保持原有粘性效果，并降低手写吸顶逻辑带来的滚动同步成本。

### 6. 富文本采用双通道展示
- 完整态：`RichText`
  - 基于 `UITextView + RichTextLayoutManager`，保留完整 HTML 展示能力。
- 收起态：`CollapsedRichTextPreview`
  - 基于 `UILabel`，使用系统尾部省略号截断。
  - 刻意放弃引用竖线与自定义列表圆点，以换取更低的测量与绘制成本。
- 外层组件：`ExpandableRichText`
  - 根据展开状态切换上述两条渲染通道。

### 7. 缓存与预热策略
- `RichTextRenderCache` 统一缓存：
  - HTML -> `NSAttributedString`
  - 宽度 + 行数约束下的布局快照
- 列表行通过 `.equatable()` 限制无关刷新。
- ViewModel 通过月份预加载减少滑动到新月份后的标记空窗期。
- `TimelineListContainer` 仅在 `bootstrapPhase == .ready && !isRefreshing` 时触发富文本预热，避免首开与刷新期抢占资源。

## 交互与视觉策略

### 1. 为什么不用重 loading 方案
- 本次问题是结构稳定性问题，不是“用户不知道系统在忙什么”的反馈问题。
- 因此不引入以下方案作为主表达：
  - 骨架屏
  - 大面积 shimmer
  - 全屏 spinner
- 这些方案虽然能遮盖数据未就绪，但会把时间线从“内容页面”降级成“等待页面”，不符合当前产品的轻量体验目标。

### 2. 静态壳层的设计要求
- 静态壳层必须满足三个条件：
  - 结构接近真实页面，避免 reveal 后大幅跳动。
  - 视觉克制，不制造额外关注点。
  - 不伪造业务结果，不提前显示“没有数据”。
- 因此当前壳层只保留：
  - 月份头部
  - 星期行和静态日历网格
  - 列表 section 节奏、时间线连接线和卡片块级占位

### 3. 刷新期反馈策略
- 刷新不是首开，因此不能再次用壳层覆盖整个页面。
- 当前刷新期只做轻提示：
  - 旧内容保留在位
  - 顶部或局部出现轻量“正在更新”反馈
- 这样可以维持用户的空间记忆，不破坏滚动上下文。

## 性能设计要点

### 1. 列表 diff 降载
- `TimelineViewModel.applySections(_:)` 只在分组结果实际变化时更新 `sections`。
- `sectionsRevision` 作为轻量 revision token，避免滚动期间对整个 section 树做深层 `Equatable` 比较。

### 2. 日历桥接刷新收窄
- 日历面板只依赖 `CalendarDependencyToken`。
- 列表 sections 变化不会直接触发整个日历桥接层重建。
- 首屏只保证当前月份 marker 就绪，相邻月份 marker 在 ready 后异步补齐。
- 月份标记使用“目标月 ± 1”预加载，降低连续翻月时的数据库抖动。

### 3. 富文本测量降载
- 折叠态不再使用完整 `UITextView` 布局。
- 截断判断与尺寸测量结果缓存到共享 `NSCache`。
- 同一 HTML、字体、颜色、宽度条件下复用布局快照，减少列表滑动中的重复测量。

### 4. 首帧稳定优先于预热命中
- 首开阶段不再显示：
  - `Color.clear`
  - 整块 `ProgressView`
  - “当日没有匹配事件”的伪空态
- 首帧稳定的依赖顺序是：
  1. 先渲染容器壳层
  2. 再等待首屏快照
  3. 最后 reveal 真内容
- 这保证即使 warmup 未命中，页面也只是停留在静态壳层，而不是结构反复变形。

## 失败与降级策略

### 1. warmup 未命中
- 表现：
  - 用户切到时间线时，看到静态壳层。
  - 首屏快照完成后再进入正式内容。
- 影响：
  - 仅影响等待时长，不影响结构稳定性。

### 2. 首屏数据为空
- 只有在 `bootstrapPhase == .ready` 后才允许展示真实空态。
- 这样可以避免“还没查完就先说没有内容”的伪空态。

### 3. 后续刷新失败
- 已有内容场景下，如果刷新失败，应保留旧数据和现有结构。
- 不允许因为一次失败回退到首开壳层，否则用户会感知为页面崩塌或重新初始化。

## 职责边界

### 1. ReadingContainerView
- 负责：
  - 二级页切换
  - `TimelineViewModel` 生命周期托管
  - 时间线 warmup 调度
- 不负责：
  - 时间线内部的日历/列表细节渲染

### 2. ReadingTimelineView
- 负责：
  - 根据是否拿到 `TimelineViewModel` 决定显示静态壳层还是正式内容
  - 承接内容查看与书籍详情回调
- 不负责：
  - 自行从环境构建状态层
  - 自行决定 warmup 时机

### 3. TimelineViewModel
- 负责：
  - 首屏快照构建
  - 日期、分类、月份 marker 的状态编排
  - 刷新与增量加载状态切换
- 不负责：
  - 具体页面动画与容器显隐

### 4. SubtabBootstrapCoordinator
- 负责：
  - 二级页级别的一次性 warmup 去重
  - 启动阶段状态跟踪
- 不负责：
  - 持有业务数据
  - 决定页面展示何种视觉形态

## 当前已知差距
- iOS 尚未暴露时间范围设置入口。
- iOS 尚未接入 item 点击跳转、复制、删除、TTS 等二级操作。
- 这些差距属于功能补齐问题，不影响当前已交付的滚动性能优化链路。

## 首开状态机

### 1. 容器阶段
- `ReadingContainerView` 创建 `TimelineViewModel`
- `SubtabBootstrapCoordinator` 触发 `.timeline` warmup
- `ReadingTimelineView` 如果尚未拿到状态源，直接展示 `ReadingTimelineBootstrapShellView`

### 2. bootstrap 阶段
- `TimelineViewModel.bootstrapPhase == .bootstrapping`
- UI 展示：
  - `TimelineBootstrapCalendarPanel`
  - `TimelineBootstrapListShell`
- 不可见状态：
  - spinner
  - 错误空态
  - 真实 section 列表

### 3. ready 阶段
- `TimelineViewModel.bootstrapPhase == .ready`
- UI 一次性切换到真实：
  - 日历 marker
  - 真实 section 列表
  - 分类筛选入口可交互

### 4. refresh 阶段
- `TimelineViewModel.isRefreshing == true`
- 保留旧内容在位
- 仅展示右上角轻提示“正在更新”
- 完成后替换内容，不回退到 bootstrap 壳层

## 时序说明

### 1. 冷启动命中 warmup
1. `ReadingContainerView` 创建 `TimelineViewModel`
2. 容器以 `.utility` 调度 `.timeline` warmup
3. `loadInitialData()` 在后台完成首屏快照
4. 用户切到时间线时，`ReadingTimelineView` 直接拿到 ready 状态并展示正式内容

### 2. 冷启动未命中 warmup
1. 用户切到时间线时，`TimelineViewModel` 仍处于 `bootstrapping`
2. `ReadingTimelineView` 展示静态壳层
3. `loadInitialData()` 完成并原子提交首屏快照
4. 页面一次性 reveal 正式内容

### 3. 页面已展示后的筛选切换
1. 用户切换分类或日期
2. `TimelineViewModel` 设置 `isRefreshing = true`
3. 旧内容留在原地，局部显示轻提示
4. 新内容返回后替换 `sections`
5. `isRefreshing` 结束，不回退到 bootstrap

## 验证口径
- 主流程验证：从“在读”切换到“时间线”后首帧先看到稳定壳层，再一次性进入正式内容，不出现碎裂感。
- 数据验证：切换日期、切换分类后，列表与日历标记同步变化。
- 交互验证：日期头保持粘性吸顶；“今”按钮能跳回当天。
- 性能验证：列表滚动中富文本收起态不卡顿，展开/收起行为保持原有交互语义。
- 降级验证：即便 warmup 完全未命中，也不出现空白页、spinner 或伪空态。
