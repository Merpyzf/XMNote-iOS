# Git 提交风格规范

目标：统一仓库提交信息的命名方式，优先复用历史提交中已经稳定存在的功能名，减少同义词漂移，保持 Android → iOS 重构交付过程中的提交整洁度与可追溯性。

## 1. 基本格式

- 提交信息必须使用中文。
- 统一格式：`type(功能模块): 动作 + 结果`
- 当前允许的 `type` 仅包括：
  - `feat`
  - `fix`
  - `refactor`
  - `chore`
  - `docs`
  - `test`
  - `build`
  - `ci`
  - `revert`
- 历史中的 `perf`、`optimize`、无括号 `feat:` 等写法属于存量历史，不作为当前推荐风格，也不纳入新规范。

推荐示例：

- `feat(书籍选择): 收口远端直返并补齐调试验证`
- `fix(时间线): 修正返回后月份回跳到最新月`
- `refactor(书摘编辑): 收敛元数据行实现并简化调试入口`
- `docs(书籍选择): 补齐对齐文档并同步术语收口`

## 2. 功能名选择原则

- 括号中的“功能模块”表达业务域或稳定功能域，不表达实现手段、文件名、技术动作或一次性页面操作。
- 优先复用历史提交中已经存在且语义贴近的名称，保持提交历史连续性。
- 复用时必须保持原有中文写法一致，不自行发明近义词。
- 只有在历史提交中找不到语义接近的名称时，才允许新增新的功能名。
- 同一功能的代码提交、文档提交、术语收口提交，默认复用同一功能名。

优先复用的历史稳定名称示例：

- `阅读日历`
- `时间线`
- `书摘编辑`
- `在读`
- `笔记查看`
- `基础设施`
- `书籍搜索`
- `书籍选择`
- `数据备份`
- `工程`
- `工程配置`

## 3. 历史功能名复用规则

### 选择流程

1. 先判断这次改动的主业务意图，而不是先看改了哪些文件。
2. 在历史提交中搜索相近功能名。
3. 若存在明确等价项，复用完全相同的名称。
4. 若不存在，再新增一个简洁、稳定、面向业务的名称。

可用搜索方式示例：

```bash
git log --pretty=format:'%s' | rg '^(feat|fix|refactor|docs|chore)\('
git log --pretty=format:'%s' | rg '书籍选择|时间线|书摘编辑'
```

### 复用优先级

- 功能名复用优先级高于“重新命名更好看”。
- `feat()`、`refactor()`、`fix()`、`docs()` 等所有带括号类型，都使用同一套功能名选择逻辑。
- 文档提交和代码提交若属于同一功能收口，默认复用同一功能名。

### 新增功能名规则

只有在历史里没有语义等价项时，才允许新增功能名。新增时必须满足：

- 以 2-6 个中文字符为主，必要时允许稍长，但不要膨胀成一句话。
- 优先业务域，不优先技术名。
- 长期可复用，不写一次性页面动作。
- 不带“组件”“交互优化”“链路调整”这类动作性尾巴，除非该词本身已经是历史稳定名称。
- 不与现有名称形成近义重复。

示例：

- 已有 `书籍选择` 时，不再新增 `选书组件`
- 已有 `时间线` 时，不再新增 `阅读时间流`
- 已有 `基础设施` 时，不再新增 `底层能力` 作为同义模块名

### 保底大类的使用边界

- `基础设施`：用于跨多个业务域的公共能力、基建或底层支持改动，且确实找不到更具体的功能域时使用。
- `工程`、`工程配置`：仅用于 Scheme、工程文件、构建配置、本地化元数据、依赖接入等仓库级治理改动。
- `规范`：仅用于仓库级协作规范、治理规则、提交流程文档等非业务代码改动。

不要把这些保底大类当成默认选项。

## 4. 提交整洁度规则

- 一个提交只承载一个逻辑变更。
- 若跨多个无关功能域，默认拆分提交。
- 若同一变更同时触达代码、文档、术语表，但服务同一功能收口，可使用同一个功能名。
- 只有在仓库级或跨域治理场景，才允许使用 `工程`、`工程配置`、`规范` 这类更宽的模块名。
- 文档收口提交优先沿用对应功能名。

推荐：

- `feat(书籍选择): 落地通用书籍选择组件并接入书摘编辑`
- `docs(书籍选择): 补齐对齐文档并同步术语收口`

不推荐：

- `feat(书籍模块): 落地通用书籍选择组件并接入书摘编辑`
- `docs(规范): 补齐书籍选择文档`

因为这会打断同一功能在历史中的连续命名。

## 5. 示例与反例

### 正例

- `feat(书籍选择): 收口远端直返并补齐调试验证`
- `refactor(书摘编辑): 收敛元数据行实现并简化调试入口`
- `fix(时间线): 修正返回后月份回跳到最新月`
- `docs(书籍选择): 补齐对齐文档并同步术语收口`
- `chore(工程配置): 修正 Scheme 引用并同步本地配置`
- `feat(基础设施): 建立加载反馈基建并统一页面加载体验`
- `fix(书籍搜索): 对齐豆瓣搜索结果展示并精简结果卡片头部`
- `refactor(弹窗): 收敛系统桥接并固化使用规范`

### 反例

- `feat(选书组件): 收口远端直返并补齐调试验证`
原因：历史里已有稳定名称 `书籍选择`，不应新造近义词。

- `refactor(BookPickerView): 重构结果类型`
原因：使用实现名代替功能域，不利于历史连续性。

- `fix(工程): 修复时间线回跳问题`
原因：明明是 `时间线` 功能问题，却使用了过宽模块名。

- `feat(书籍选择): 收口远端直返并补齐调试验证；修复时间线回跳`
原因：把两个无关功能域塞进一个提交，破坏提交整洁度。

- `optimize(时间线): 优化滚动性能`
原因：`optimize` 属于历史存量写法，不是当前允许的提交类型。

- `feat: 搭建书籍选择能力`
原因：缺少括号中的功能模块，不符合当前格式。
