# 颜色系统优化规范

用于统一 XMNote 的颜色设计、编码与验收标准，建立「HIG 语义优先 + 品牌表达保留」的可持续体系。

## 1. 背景与目标

当前项目已经具备 `DesignTokens.swift` 统一令牌能力，但基础语义层仍以自定义色值为主，导致以下问题：
- 与 iOS 系统语义色有偏差，深浅色/高对比度维护成本上升。
- 部分强调色被用于正文文本，存在可读性风险。
- 局部仍有业务文件直接写颜色，难以全局治理。

本规范目标：
- 建立分层颜色架构，明确“系统语义色”和“品牌业务色”边界。
- 提升可访问性与跨版本稳定性。
- 为后续渐进迁移提供可执行路线与验收闸门。

## 2. 适用范围

- 适用于 `xmnote/` 全量 UI 代码（SwiftUI/UIKit）。
- 默认覆盖浅色、深色、高对比度三类场景。
- `xmnote/Utilities/DesignTokens.swift` 作为颜色单一真相源。

## 3. 颜色分层模型（强制）

### L0 平台语义层（System Semantic）
- 直接来源：Apple 系统语义色。
- 核心颜色：
  - Text: `label`, `secondaryLabel`, `tertiaryLabel`, `quaternaryLabel`
  - Background: `systemBackground`, `secondarySystemBackground`, `systemGroupedBackground`, `secondarySystemGroupedBackground`
  - Fill: `systemFill`, `secondarySystemFill`, `tertiarySystemFill`, `quaternarySystemFill`
  - Separator: `separator`, `opaqueSeparator`
  - Interaction: `link`, `tintColor`
- 目的：平台一致性、自动适配、降低维护成本。

### L1 产品语义层（Product Semantic）
- 基于 L0 封装项目级语义，不直接暴露外观色值。
- 示例命名：
  - `semanticTextPrimary/Secondary/Tertiary`
  - `semanticBackgroundPage/Card`
  - `semanticSeparator/SeparatorStrong`
  - `semanticTint/semanticLink`

### L2 品牌与业务主题层（Brand & Feature Theme）
- 保留产品识别与业务编码色：
  - `brand*`
  - `heatmap*`
  - `readCalendar*`
- 只用于品牌强调和业务可视化，不替代通用文本/背景语义。

### L3 组件语义别名层（Component Alias，可选）
- 仅跨模块复用组件可定义组件别名（如 TopBar/Card/Chart）。
- 页面私有样式不得上升为全局基础令牌。

## 4. 命名与使用规则（强制）

### 4.1 命名规则
- 平台映射统一 `semantic*` 前缀。
- 品牌/业务主题统一 `brand*`、`readCalendar*`、`heatmap*`。
- 禁止无语义命名：`gray1`、`lightBlack`、`textColor2`。

### 4.2 使用规则
- 正文文本默认使用 `semanticTextPrimary/Secondary/Tertiary`。
- 页面背景/容器背景默认使用 `semanticBackground*`。
- 边框/分隔默认使用 `semanticSeparator*`。
- 交互强调默认使用 `semanticTint`（必要时桥接 `tintColor`）。
- 禁止在业务文件直接新增 `Color(hex:)` 或裸 `UIColor(red:...)`。

### 4.3 禁止误用（重点）
- `brand`、`feedbackWarning`、`feedbackError` 不得直接作为长文本正文前景色。
- 状态信息必须双通道表达：颜色 + 文案/图标，不允许仅靠颜色区分。

## 5. `DesignTokens.swift` 重构指引

### 5.1 结构分区建议
- `SystemSemanticColor`
- `ProductSemanticColor`
- `BrandColor`
- `FeatureThemeColor`
- `ColorHelpers`（仅保留必要构造器）

### 5.2 兼容迁移策略
- 旧 token 先保留为兼容别名，内部转发到新 `semantic*`。
- 新代码一律使用新语义 token。
- 迁移完成后再移除旧 token。

### 5.3 推荐映射
- `textPrimary -> semanticTextPrimary`
- `textSecondary -> semanticTextSecondary`
- `textHint -> semanticTextTertiary`
- `windowBackground -> semanticBackgroundPage`
- `contentBackground -> semanticBackgroundCard`
- `cardBorder/surfaceBorderDefault -> semanticSeparator`
- `surfaceBorderStrong -> semanticSeparatorStrong`

## 6. 分阶段迁移路线

### Phase A：地基搭建
- 在 `DesignTokens` 完成 L0/L1/L2 分层。
- 旧 token 转发新语义，保证编译和 UI 稳定。

### Phase B：高风险点收敛
- 优先替换“强调色做正文”的场景。
- 聚焦错误提示、重试文案、辅助说明等可读性高风险区域。

### Phase C：全仓收口
- 清理业务文件颜色硬编码。
- 全量替换基础语义旧 token 到新语义层。

### Phase D：治理固化
- 引入静态校验闸门与 PR 检查单。
- 调试页持续作为语义教学与回归基线。

## 7. 验收标准

### 7.1 可访问性
- 正文文本对比度 `>= 4.5:1`。
- 大字号文本或强调标题对比度 `>= 3.0:1`。
- 非文本关键边界（按钮/分隔）在深浅色可辨识。

### 7.2 主题一致性
- 浅色与深色模式层级稳定，无“反转感”。
- 高对比度下信息层级仍清晰。

### 7.3 业务表达
- 品牌识别（brand）保持一致。
- 阅读热力图、阅读日历等业务主题色语义不退化。

## 8. 工程闸门与协作要求

### 8.1 静态校验建议
- 业务目录禁用新增 `Color(hex:)`。
- 检测禁用 token 作为正文前景色（`brand/feedback*`）。

### 8.2 PR 检查单
- 是否使用语义 token，而非外观色值。
- 是否新增颜色误用（背景色做文本色等）。
- 是否补充浅色/深色截图证据（关键页面）。

### 8.3 调试基线
- 使用 `SystemColorsTestView` 作为系统语义色案例基线。
- 用于评审“该颜色是否应迁入系统语义层”。
