# 页面状态基建与开发模式

## 1. 背景与目标

本项目页面状态方案的目标是：
- 在 iOS 原生语义下稳定实现 `push -> pop` 状态连续性与 scene 级恢复。
- 明确状态 owner 与生命周期边界，避免“谁在改状态”不清楚导致的误判。
- 控制复杂度，不引入超出业务必要的“通用保活框架”。

非目标：
- 不追求像素级滚动位置跨重建恢复。
- 不通过统一滚动宿主包装所有页面。
- 不在无共性根因时上升到全局抽象。

## 2. 基建总览（当前实现）

当前基建由以下部分组成：
- `SceneStateStore`：scene 级轻量快照聚合与持久化出口。
- `AppSceneSnapshot`：根 tab、根导航路径、容器子 tab 与高价值页面语义锚点。
- `@SceneStorage("xmnote.scene.snapshot")`：单 scene 的持久化落盘。
- `NavigationPath.CodableRepresentation`：导航栈恢复。
- `KeepAliveSwitcherHost`：仅负责切页存活与体验，不承载数据恢复策略。

核心原则：
- 恢复与持久化是“语义锚点恢复”，不是“完整 UI 内存镜像恢复”。
- 数据真相在 Repository，页面快照只保存 UI 选择态与导航语义态。

## 3. 状态分层与 Owner 规则（强约束）

### 3.1 会话内状态（View 层）
- 载体：`@State`、`@Observable` ViewModel。
- 生命周期：视图树存活期间。
- 示例：搜索输入、当前筛选、当前选择项、页面局部弹层状态。

### 3.2 Scene 恢复状态（基建层）
- 载体：`SceneStateStore + SceneStorage`。
- 生命周期：当前 scene 跨前后台、跨进程重建。
- 示例：当前 tab、NavigationPath、时间线月份/分类、阅读日历月份/年份/模式。

### 3.3 用户偏好状态（设置层）
- 载体：`@AppStorage`。
- 生命周期：用户级持久化。
- 示例：时间线展示范围、功能开关偏好。

### 3.4 业务数据状态（数据层）
- 载体：Repository / Database / Network。
- 生命周期：由数据源定义。
- 规则：ViewModel 禁止绕过 Repository 直连底层数据设施。

## 4. 本次会话已落地的关键机制

### 4.1 根恢复链路
- `ContentView` 在 `task(id: appState.dataEpoch)` 中执行 scene 快照恢复。
- `MainTabView` 在 `sceneStateStore.isRestored` 后一次性 bootstrap：
  - 恢复 `selectedTab`、`searchQuery`
  - 恢复五个根 `NavigationPath`

### 4.2 容器与页面接入
- 根容器恢复子 tab：
  - `ReadingContainerView` / `BookContainerView` / `NoteContainerView`
- 高价值页面恢复语义锚点：
  - `ReadingTimelineView`：`selectedDate`、`displayedMonthStart`、`selectedCategory`
  - `ReadCalendarView`：`pagerSelection`、`selectedDate`、`displayMode`、`selectedYear`
  - `BookSearchView`：`query`、`selectedSource`
  - `ContentViewerView`：按 `source` 恢复 `selectedItemID`

### 4.3 本次高风险修复（必须继承）
- 阅读日历写回竞态修复：
  - 恢复未完成前禁止写回 snapshot（门闩机制）。
- 阅读日历快照一次性消费：
  - 首轮恢复后清空 pending snapshot，避免 reload 回跳旧月份。
- 时间线分类 Codable 兼容修复：
  - 持久化编码使用稳定 code，不再绑定中文展示文案；解码兼容历史值。
- 导航路径静默失效修复：
  - `path.codable == nil && path 非空` 时不覆盖已有快照，并输出告警日志。

## 5. 新页面接入标准流程

接入准入条件（先判断再接入）：
- 必须是跨 scene 生命周期仍影响用户返回认知的高价值状态。
- 必须先完成 owner、写入点、生命周期时机的事实核对。
- 单页面现象先按单点修复；未证明共性前禁止推广为公共抽象。

满足准入条件后，按以下步骤接入：

1. 定义快照
- 在 `AppSceneSnapshot` 下新增最小语义快照结构。
- 仅保存“影响用户返回认知”的锚点，禁止保存临时 UI 噪声状态。

2. 扩展 `SceneStateStore`
- 增加 `updateXxx(...)` 与读取入口。
- 写入逻辑必须具备“无变化不持久化”特性。

3. 页面恢复时序
- 页面首次启动必须等待 `sceneStateStore.isRestored == true`。
- 恢复顺序固定为：`apply snapshot -> loadIfNeeded -> enable persist gate`。

4. 页面写回策略
- 只在关键状态变化时写回（`onChange`）。
- 恢复阶段必须 gate 掉写回，防止默认态覆盖旧快照。

5. 导航恢复约束
- 所有进入 `NavigationPath` 的 route payload 必须 `Codable`。
- 若某路径元素不能编码，保持 last known good，并记录日志告警。

## 6. 禁止项（红线）

- 禁止把展示文案当持久化协议值（例如中文标题直接参与 Codable 协议）。
- 禁止在恢复未完成时写回快照。
- 禁止遇到单页面问题就上升为“全局保活基建问题”。
- 禁止引入通用滚动保活包装替代真实 owner 归因。
- 禁止无日志地静默吞掉恢复失败。
- 禁止在未证明共性前推广抽象到所有页面。

## 7. 故障排查手册

### 7.1 现象：返回后状态没保留
- 检查 `isRestored` 是否完成后才启动页面恢复。
- 检查是否存在恢复阶段写回覆盖（看 persist gate）。
- 检查快照字段是否真的被写入（查看 `SceneStateStore` 日志）。

### 7.2 现象：阅读日历回跳到旧月份
- 检查 pending snapshot 是否一次性消费。
- 检查设置刷新 / reload 是否重新应用了旧 snapshot。

### 7.3 现象：导航栈恢复丢失
- 检查 path 中 route/payload 是否全部 `Codable`。
- 检查是否触发了 non-codable path 告警日志。

### 7.4 现象：冷启动恢复为空
- 检查 `dataEpoch` 是否变化导致快照重置（这是预期保护行为）。
- 检查 `SceneStorage` 数据是否可解码、版本是否兼容。

## 8. Android Compose 对照

对应关系（意图对齐，不是机制照搬）：
- Compose `ViewModel + StateFlow` ↔ iOS `@Observable ViewModel`
- Compose `remember` ↔ iOS `@State`
- Compose `rememberSaveable` ↔ iOS `SceneStorage + AppStorage`（按状态语义拆分）
- Compose `NavController back stack` ↔ iOS `NavigationStack + NavigationPath`
- Compose `LazyListState` ↔ iOS 默认 `List/ScrollView` 返回行为 + 语义锚点恢复（非像素镜像）

关键差异：
- iOS 当前方案以“语义锚点”恢复为主，不承诺跨重建像素级滚动位置重现。
- 已接入页面由 scene 级快照统一编排；新页面按准入条件逐步接入，避免无证据扩散。

## 9. 验收清单（按执行策略分层）

默认必做（与项目执行策略一致）：
- 编译通过：`xcodebuild -project XMNote.xcodeproj -scheme xmnote -destination 'platform=iOS Simulator,name=iPhone 17 Pro' build`

按需执行（仅当用户明确要求测试/验收时）：
- 启动通过：`build_run_sim`
- 行为验收：
  - `push -> pop` 后关键状态保持
  - 冷启动后关键状态恢复
  - 数据 epoch 变化后旧快照被安全丢弃
- 日志验收：
  - 恢复失败有可定位日志
  - non-codable path 有明确告警
